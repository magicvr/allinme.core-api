# 阶段一 1A：运行基础开发计划

## 1. 目标

将当前仅有 `GET /healthz` 的进程扩展为可配置、可迁移、可重置、可诊断的 SQLite 服务骨架，为认证和订单阶段提供稳定基础。本计划完成后仍不实现账号登录、订单业务、附件或页面配置。

本计划对应 [实施路线阶段一](../../06-implementation-roadmap.md#2-阶段一运行基础已实现)。本 checklist 全部完成即视为路线图阶段一完成；页面模块的 readiness 检查仍在阶段六扩展。由于角色账号和订单表分别属于后续阶段，本轮只实现通用 migration/seed/reset 框架和基础 seed 元数据；角色 seed 在阶段二启用，订单及关键业务状态 seed 在阶段三启用。

## 2. 已确定选择

| 决策 | 本轮选择 | 理由 |
|---|---|---|
| SQLite driver | 纯 Go、无 CGO driver；实施时固定具体模块版本 | 降低 Windows、CI 和发布环境门槛 |
| SQLite 连接 | driver DSN/connector 对每个物理连接初始化 pragma；阶段一 `MaxOpenConns=1`、`MaxIdleConns=1` | 避免在池中只配置到偶然取得的连接 |
| migrations | `internal/store/migrations/*.sql` + `go:embed` + 轻量 runner | 当前只需线性、事务化迁移，不引入外部 CLI |
| schema 版本 | SQLite `PRAGMA user_version` | 单数据库 demo 足够，避免先建迁移元数据表 |
| seed 版本 | 独立 `seed_versions` 表，按 seed group 记录版本 | 与 schema 版本解耦，阶段二/三可增量追加 seed |
| migration 时机 | 只由 `cmd/admin migrate/reset` 执行；API 启动绝不自动 migrate | 未迁移部署可明确表现为 not ready，避免启动时隐式改库 |
| 开发命令 | 单入口 `cmd/admin`，子命令 `migrate`、`seed`、`reset` | 共享配置与装配，避免三个二进制重复代码 |
| 配置 | `APP_ENV`、`PORT`、`DATA_DIR`，经 `internal/config` 一次解析和校验 | 保留现有 `PORT`，数据路径可安全派生和测试注入 |
| readiness | `GET /readyz` 检查数据库可访问且 schema 为最新 | 页面模块尚未实现，本阶段不检查页面 |
| 错误响应 | `internal/httpapi` 统一 envelope 与集中写入函数 | 为后续 endpoint 提供唯一 transport 错误出口 |
| 日志 | 标准库 `log/slog`，由 app 注入 httpapi/store/admin | 延续当前实现并允许测试捕获，禁止包内使用全局 logger |

具体 SQLite driver 是本轮唯一需要新增的运行依赖。实施前读取当前模块文档，确认 Go 1.26、Windows、context、事务、URI 打开模式及每物理连接 DSN/connector pragma 支持，再更新 `go.mod`/`go.sum`。选型证据必须记录 driver 名称/版本、官方文档 URL 和逐项结论，不能只写“已确认”。

## 3. 目标结构

```text
cmd/
  api/main.go
  admin/main.go
internal/
  app/                 # API/admin 共享装配和资源关闭
  config/              # 环境变量解析、默认值和模式校验
  httpapi/             # handler、中间件、错误 envelope、health/ready
  store/               # SQLite 打开、pragma、迁移、事务与状态检查
    migrations/*.sql
```

测试使用 `t.TempDir()` 下的数据库文件，不使用开发数据库。共享测试 helper 只在至少两个包确有重复后提取，不预建通用 testkit。

## 4. 配置契约

阶段一冻结以下环境变量与派生规则，不增加 `HTTP_ADDR`、`LISTEN_ADDR` 或任意 `DATABASE_PATH`：

| 环境变量/派生值 | 开发默认 | 生产要求 |
|---|---|---|
| `APP_ENV` | `development` | 必须显式为 `production` |
| `PORT` | `8080`，监听地址派生为 `:<PORT>` | 显式配置；十进制 `1..65535` |
| `DATA_DIR` | `./data` | 必须是显式绝对路径 |
| SQLite 主文件 | `<DATA_DIR>/allinme.db` | 固定派生，不接受请求或独立环境变量覆盖 |
| reset 删除集合 | `allinme.db`、`allinme.db-wal`、`allinme.db-shm` | 仅 development 可执行 |
| request ID header | `X-Request-ID` | 固定 |

配置解析失败必须在监听端口或打开/修改数据库前终止。路径解析后必须确认数据库主文件及 sidecar 均直接位于规范化 `DATA_DIR` 内；reset 不删除整个数据目录。日志不得输出未来的密钥、密码或完整 Authorization header。

API 以“必须已存在”的 URI 模式打开数据库，不创建缺失文件；admin migrate/reset 才使用“允许创建”模式。reset 前对规范化 `DATA_DIR`、数据库主文件和 sidecar 执行 `Lstat`/等价 Windows 检查，拒绝符号链接、junction 或可检测的 reparse point；删除目标只来自程序固定派生的三个标准文件名，不接受调用方输入参与路径或名称匹配。

首个 migration 创建 `seed_versions`：`name TEXT PRIMARY KEY`、`version INTEGER NOT NULL`、`applied_at TEXT NOT NULL`。schema 迁移只更新 `PRAGMA user_version`；seed runner 按 `name` 独立比较，在同一事务内应用 seed 并以 UTC RFC 3339 更新时间。本轮基础 seed group 固定为 `runtime`，版本为 `1`；数据库记录高于 runner 支持版本时拒绝执行。

## 5. 实施批次

### 批次 A：配置与装配边界

1. 为 development/production、监听地址和数据库路径编写表驱动测试。
2. 测试冻结 `APP_ENV`、`PORT`、`DATA_DIR` 的默认值、合法范围、生产要求和数据库/sidecar 派生规则。
3. 实现 `internal/config`，由 `cmd/api` 只负责读取环境、组装应用和处理退出码。
4. 引入 `internal/app` 管理 store、handler 和关闭顺序；保留当前优雅停机行为。

聚焦验证：`go test ./internal/config ./internal/app`。

### 批次 B：SQLite 与 migrations

1. 选定并固定纯 Go SQLite driver，使用 `database/sql` 作为业务边界。
2. 通过 driver DSN/connector 为每个物理连接设置 `foreign_keys=ON`、`busy_timeout=5000`、`journal_mode=WAL`；不使用一次性的裸 `db.ExecContext("PRAGMA ...")` 假设整个连接池已配置。
3. 阶段一固定 `MaxOpenConns(1)` 与 `MaxIdleConns(1)`；成功 `PingContext` 后在专用 `*sql.Conn` 上读取三项 pragma，任一不符则打开失败。未来放开连接池必须先增加多连接验证。
4. 实现嵌入 SQL 的有序、事务化 migration runner，拒绝缺号、重复版本和数据库版本高于二进制。
5. 首个 migration 只创建 `seed_versions`，不提前创建账号/订单领域表。
6. 提供通用事务 helper，并测试回调失败时数据回滚。
7. store readiness 使用类型化状态，不在 handler 比较错误字符串：`database_missing`、`database_unavailable`、`schema_uninitialized`、`schema_outdated`、`schema_too_new`、`ready`。`database_missing` 仅表示规范化目标文件不存在；文件存在但无法打开、权限不足、锁等待超时、损坏或其他 I/O/driver 错误均为 `database_unavailable`。
8. API 使用可重试 readiness probe：数据库缺失或暂时不可打开不阻止 HTTP 进程启动，也不创建文件；每次探测都重新检查文件并允许重新打开/刷新状态，不永久缓存首次失败。admin 完成 migrate 后，后续探测可在不重启 API 的情况下转为 ready。
9. API 不执行 migration；不存在、`user_version=0`、低于当前版本和高于当前版本分别映射为内部状态，对外统一为 `NOT_READY`。首版最新版本为 v1，无法构造真实 `0 < version < latest` 数据库；本阶段只对版本分类函数单元测试 `schema_outdated`，真实低版本数据库集成测试延后到 migration v2。

聚焦验证：`go test ./internal/store -count=1`；迁移测试覆盖空库、重复运行、失败回滚、版本过新和进程重开。

### 批次 C：migrate/seed/reset 命令

1. 实现 `cmd/admin migrate`，成功后输出结构化摘要且重复运行无副作用。
2. 实现基于 `seed_versions` 的 seed runner；本轮只写入 `runtime=1`，不创建未来角色或订单数据。
3. 实现 `cmd/admin reset`：仅 development 可用，且要求 API 进程已停止；关闭 admin 自有连接后删除数据库相关文件，再执行 migrate + seed。
4. production 模式调用 reset 必须在任何破坏性操作前失败；目标文件被其他进程占用或无法删除时安全失败，不继续 migrate/seed，也不把部分清理写成成功。

聚焦验证：命令逻辑以函数级测试覆盖；`go run ./cmd/admin -- <subcommand>` 仅做本地 smoke，不作为唯一证据。

### 批次 D：HTTP 运行面

1. 把 `NewHandler` 改为显式依赖，保留 `GET /healthz` 的响应契约。
2. 新增 `GET /readyz`：数据库可访问且 schema 最新时返回 `200` 与 `{"status":"ready"}`；否则返回 `503` 与统一错误 envelope。
3. API 启动不自动 migrate；未迁移库可以启动，但 `/readyz` 必须为 `503`，`/healthz` 保持 `200`。
4. request ID 只接受匹配 `[A-Za-z0-9][A-Za-z0-9._-]{0,63}` 的入站值；空值或非法值生成 `req_` 加 32 位小写十六进制随机值。
5. 每个响应都回写最终 `X-Request-ID`；错误体 `requestId` 与响应 header、访问日志必须一致。
6. 中间件顺序固定为 request ID → access log → recovery → handler；response writer 跟踪是否已提交响应。下游未提交时 panic 转为完整 `500 INTERNAL_ERROR`；已提交时不重写响应，只记录 panic、保活并让 access log 反映可观察状态。
7. `httpapi` 接收注入的 `*slog.Logger`，不调用全局默认 logger；测试使用内存 handler 断言结构化属性。
8. readiness probe 贯穿 request context，并使用明确短超时；取消、超时或 store/app 已关闭时安全返回 not-ready，不泄露资源、不 panic、不无限阻塞。
9. `/readyz` 只接受 `GET`；其他 method 返回 `405`、`Allow: GET` 和 `METHOD_NOT_ALLOWED` envelope。
10. 冻结本阶段错误 envelope `{"error":{"code","message","requestId"}}`，机器码只包含 `NOT_READY`、`METHOD_NOT_ALLOWED` 与 `INTERNAL_ERROR`。

聚焦验证：`go test ./internal/httpapi -count=1`，覆盖 method/path、health 与数据库状态解耦、ready 成功/失败、context 取消/超时、关闭后调用、request ID 透传/生成、panic 提交前后语义和安全错误体。

### 批次 E：集成与文档收敛

1. 在 `internal/app` 使用临时数据目录做集成测试，验证 migrate 后 ready、数据库缺失/未迁移/损坏/版本过新时 not ready、重开后状态保留；逻辑不得只存在于 `main`。真实低版本数据库用例在 migration v2 出现后加入。
2. 更新 [当前 HTTP API](../../03-http-api.md)，把 `/readyz` 从目标草案迁入当前规范。
3. 收缩 [目标 HTTP API](../../03-http-api-target.md) 的 readiness draft，冻结本阶段实际错误 envelope 与机器码。
4. 将 [验证矩阵](../../04-validation.md) 中 SQLite、seed/reset、readiness 改为 `enabled: yes`，记录实际测试包或命令。
5. 更新架构、路线、CHANGELOG 和 checklist 证据。

最终验证：

```sh
go test ./...
go vet ./...
go test -race ./...
```

若 race 在环境中不可运行，必须在 checklist 记录具体原因，不得标记通过；替代证据至少包含 `go test ./... -count=1` 和关键共享状态/并发路径说明。

## 6. 完成标准

- 全新目录可 migrate、seed、启动 API，并使 `/healthz` 与 `/readyz` 返回成功；
- 数据库重开后 schema/seed 状态保留，重复 migrate/seed 无副作用；
- API 启动绝不自动 migrate；未迁移库的 `/readyz` 为安全 `503`，`/healthz` 为 `200`；
- production reset 在删除文件前失败，development reset 可恢复同一基础状态；
- 数据库不可用或 schema 不匹配时 `/readyz` 返回安全的 `503`，`/healthz` 仍为 `200`；
- 配置、store、admin 命令与 HTTP 测试使用隔离临时资源；
- SQLite、seed/reset、readiness 三项门禁已启用并记录证据；
- 当前/目标 API、架构、路线和 CHANGELOG 与实现一致；
- 本 checklist 全部完成即视为路线图阶段一完成，阶段六只扩展页面 readiness。

## 7. 非目标

- 用户表、密码哈希、JWT、session 和角色 seed；
- 订单、退款、附件、看板或页面配置表；
- 通用 repository 框架、缓存、后台任务或多数据库支持；
- 生产级备份、在线迁移、集群 readiness 或 Kubernetes 配置；
- 为尚未实现的业务错误一次性冻结完整错误码全集。

## 8. 风险与回退

| 风险 | 控制 |
|---|---|
| SQLite driver 与 Go 1.26 不兼容 | 引入前验证官方版本说明并运行最小 store 测试 |
| reset 误删非目标文件 | 只接受配置生成的数据库路径，校验 development，并测试删除前失败路径 |
| migration 半执行 | 每个版本单事务；失败保持原 `user_version` |
| 多进程同时迁移 | 本 demo 启动模型要求部署前单独 migrate；并发行为由 busy timeout 明确失败 |
| readiness 泄露内部信息 | 外部只返回稳定状态与 request ID，详细错误仅进入安全日志 |
| 验证矩阵超前要求乐观锁 | SQLite 门禁只覆盖通用 store；乐观锁移到订单阶段启用 |
| env 或路径歧义导致误删 | 表驱动测试冻结 `PORT`/`DATA_DIR` 派生规则，reset 只删固定数据库文件集合 |
| 连接池产生未配置 pragma 的连接 | DSN/connector 对每个物理连接初始化；阶段一限制单连接并读取验证 |
| Windows link/reparse 绕过 reset 边界 | 规范化并检查目录与目标，拒绝链接/junction/reparse，basename 严格 allowlist |
| API 持有 SQLite/WAL 句柄导致 reset 不完整 | reset 明确要求 API 停止；文件占用/删除失败时终止且不继续 migrate/seed |

实施清单见 [`0001-2026-07-12-checklist.md`](./0001-2026-07-12-checklist.md)。
