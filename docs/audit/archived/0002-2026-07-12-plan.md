# 阶段二：认证授权开发计划

## 1. 目标

在阶段一运行基础上实现本地账号登录、短期 JWT Bearer、SQLite 可撤销 session 和角色授权，为后续订单、退款、附件与页面 API 提供统一认证主体。本阶段完成后交付 `POST /api/v1/auth/login`、`GET /api/v1/auth/me` 和 `POST /api/v1/auth/logout`，但不实现账号管理、订单或页面入口。

本计划对应 [实施路线阶段二](../../06-implementation-roadmap.md#3-阶段二认证授权已实现)，遵循 [认证 ADR](../../decisions/0002-local-jwt-with-revocable-sessions.md) 和 [当前 HTTP API](../../03-http-api.md#认证-api)。配套 checklist 已全部完成并于 2026-07-12 经用户确认归档。

## 2. 已确定边界

| 决策 | 本阶段选择 | 约束 |
|---|---|---|
| 密码哈希 | `golang.org/x/crypto/bcrypt v0.54.0`，cost 固定为 `11` | 密码长度按 UTF-8 编码后的 byte 数校验为 12..72 bytes；明文只存在于请求解码和命令输入的最短生命周期 |
| JWT | `github.com/golang-jwt/jwt/v5 v5.3.1`，HS256 | parser 固定算法、issuer、audience，并要求 `sub`、`exp`、`iat`、`jti` 和 role |
| 签名密钥 | `JWT_SIGNING_KEY`，至少 32 bytes | development 与 production 均显式提供；不得使用仓库内默认值、启动时随机值或日志输出 |
| token 生命周期 | 固定 15 分钟，UTC 时间 | 不实现 refresh token；JWT `exp`、未来 `iat` 与 session `expires_at` 统一使用 30 秒时钟偏差 |
| issuer / audience | `allinme.core-api` / `allinme.schema-ui` | 作为服务端常量，不接受请求覆盖 |
| session | 每个 JWT `jti` 对应一条 SQLite session | 允许同一用户并存多条未撤销 session；认证要求 `session.user_id == claims.sub`，并校验用户禁用状态和当前角色 |
| 登出 | 撤销当前 session | 首次成功返回 `204`；旧 token 后续请求返回统一未认证错误 |
| 角色 | `viewer`、`operator`、`approver`、`admin` | 本阶段只提供可复用策略与测试探针，不新增账号管理或业务 endpoint |
| 演示账号 | development seed 固定创建 `viewer`、`operator`、`approver`、`admin` | username 与 role 同名；共享演示密码从 `DEMO_ACCOUNT_PASSWORD` 读取；production 不创建演示账号 |
| production 首账号 | 一次性 `cmd/admin bootstrap-admin` | 仅 production 且 `users` 表为空时创建首个 admin；不是通用账号管理接口 |
| 登录限流 | 单进程内按规范化用户名与客户端 IP 组合限流 | 不信任转发 header；状态不持久化，适配当前单实例 demo 边界 |

密码库与 JWT 库已在批次 A 读取当前官方文档并固定版本。2026-07-12 本机单次 bcrypt hash 实测 cost 10/11/12 分别约为 51/105/209 ms，因此固定 cost `11`。JWT v5 parser 使用 `WithValidMethods`、`WithExpirationRequired`、`WithIssuer`、`WithAudience`、`WithIssuedAt`、`WithTimeFunc` 和 `WithLeeway`；其中 `WithIssuedAt` 只校验存在的 `iat`，因此自定义 claims validation 还要求 `sub`、`iat`、`jti` 和 role 非空且 role 合法。

12-byte 密码下限是本地 demo 的明确安全基线，不是为了与 72-byte 上限对称。ASCII 密码至少需要 12 个字符；多字节密码仍按 UTF-8 byte 数执行同一边界，不额外声称字符复杂度。若系统脱离 demo 定位，应重新评估密码策略、泄露密码检查和 MFA，而不是只调整该数字。

## 3. 配置与安全契约

阶段二在现有 `APP_ENV`、`PORT`、`DATA_DIR` 上增加：

| 环境变量 | development | production |
|---|---|---|
| `JWT_SIGNING_KEY` | API 启动必须显式配置，至少 32 bytes | API 启动必须显式配置，至少 32 bytes |
| `DEMO_ACCOUNT_PASSWORD` | 执行 auth seed 时必须显式配置，长度 12..72 bytes | 忽略且禁止创建演示账号 |
| `BOOTSTRAP_ADMIN_USERNAME` | 不使用 | `bootstrap-admin` 必须显式配置，执行相同 username 规范化且结果非空 |
| `BOOTSTRAP_ADMIN_PASSWORD` | 不使用 | `bootstrap-admin` 必须显式配置，UTF-8 编码后长度 12..72 bytes |

配置采用四个显式加载入口，不要求所有命令加载同一组 secret：基础 `config.LoadBase`、API `config.LoadAPI`、development seed/reset `config.LoadDemoSeed` 和 production 首账号 `config.LoadBootstrapAdmin`。入口先解析 admin 子命令，再选择配置加载函数；不得继续在 `cmd/admin` 启动时无条件调用单一 `config.Load`。

- `cmd/api` 使用 `LoadAPI` 加载基础运行配置与 `JWT_SIGNING_KEY`，在监听端口和打开数据库前完成校验；
- `cmd/admin migrate` 使用 `LoadBase`，不读取 JWT 密钥或演示密码；
- `cmd/admin seed` 先用 `LoadBase` 判定环境，development 再用 `LoadDemoSeed` 加载并校验 `DEMO_ACCOUNT_PASSWORD`，production 不读取该值且不运行 auth demo seed；
- `cmd/admin reset` 先按 `LoadBase` 执行 production 拒绝，development 再用 `LoadDemoSeed`，并在打开数据库、获取锁或删除文件前完成演示密码校验。
- `cmd/admin bootstrap-admin` 先用 `LoadBase` 要求 production，再用 `LoadBootstrapAdmin` 校验 username/password；所有 secret 校验在打开数据库前完成。

所有认证失败对外采用稳定、安全的消息。未知用户名、错误密码和禁用账号必须返回相同状态、机器码与消息，并为未知用户执行进程启动时预生成的固定 dummy hash 比较以降低用户名枚举的时序差异；dummy hash 使用与真实账号相同的冻结 bcrypt cost，不在每次请求中重新生成。响应、访问日志和内部错误日志不得包含密码、密码哈希、JWT、签名密钥、完整 Authorization header 或 session token ID。

登录 JSON body 上限固定为 4 KiB，使用严格解码：要求 `username` 和 `password`、拒绝未知字段、多余 JSON 值和非 JSON Content-Type。用户名规范化固定为 `strings.TrimSpace` 后 `strings.ToLower`，数据库唯一性、限流 key 与查询使用同一结果；不增加 Unicode NFKC 等隐式变换。密码不 trim、不变换 Unicode，长度一律使用 `len([]byte(password))` 按 UTF-8 byte 数校验为 12..72；未知用户与已知用户在 password compare/dummy compare 前经过完全相同的长度拒绝路径。

文档只说明如何设置 `DEMO_ACCOUNT_PASSWORD`，运行时输出和日志永不打印密码值。该规则取代领域模型中“提示演示密码”的歧义：实现收敛时把上游表述改为提示环境变量配置方式，而不是提示具体密码。

登录限流首版固定为每个“规范化用户名 + 客户端 IP”组合 1 分钟最多 5 次失败。客户端 IP 只从 `http.Request.RemoteAddr` 通过 `net.SplitHostPort` 提取并使用 `net/netip` 解析为规范字符串，同一 IP 的不同源端口必须得到同一 key；IPv4、IPv6 和 malformed address 均需有确定性测试，无法安全提取 IP 时使用固定的 `unknown` 分区，不回退到包含端口的原字符串。不得信任 `X-Forwarded-For`、`Forwarded` 等 header。

限流检查发生在密码校验之前：窗口内前 5 次失败执行正常认证，第 6 次及后续请求直接返回 `429 RATE_LIMITED`，即使携带正确密码也不绕过；窗口结束后才恢复认证。`Retry-After` 使用当前窗口剩余秒数向上取整，最小为 `1`。成功登录只在未被限流且认证成功时清除该组合的失败计数；注入 clock 并定期清理过期 bucket。

只有未知用户、错误密码和禁用账号增加失败计数。`400`、`415`、context 取消/超时、store 或其他内部错误均不计数；已返回的 `429` 不重复增加计数或延长窗口。

limiter 最多保存 4096 个活跃 bucket，每个规范 IP 最多占用 32 个 username bucket；`unknown` 视为一个 IP 分区。创建新 key 前先同步清理所有已过期 bucket。单 IP 达到 32 个 key 后，该 IP 的新 username 直接返回 `429 RATE_LIMITED` 与 `Retry-After: 1`，不影响其他 IP，也不插入 map。

若全局清理后仍达到 4096，仅可淘汰最久未访问且失败数小于 5 的未封禁 bucket；失败数已达到 5 的受限 bucket在窗口结束前不可淘汰。若所有 bucket 都受保护，则新 key fail-closed 返回 `429` 与 `Retry-After: 1`。单 IP 不能独占全局容量，高基数测试必须证明其他 IP 的已有与新登录不会仅因某一 IP 的用户名洪泛被全局拒绝。

## 4. 数据模型与 migration v2

新增 `internal/store/migrations/0002_auth.sql`，只包含认证领域表与索引：

### `users`

- `id TEXT PRIMARY KEY`；
- `username TEXT NOT NULL UNIQUE`，保存规范化用户名；
- `password_hash TEXT NOT NULL`；
- `role TEXT NOT NULL`，约束为四个内置角色之一；
- `disabled_at TEXT NULL`；
- `created_at TEXT NOT NULL`、`updated_at TEXT NOT NULL`。

### `sessions`

- `id TEXT PRIMARY KEY`；
- `user_id TEXT NOT NULL REFERENCES users(id)`；
- `token_id TEXT NOT NULL UNIQUE`，对应 JWT `jti`；
- `expires_at TEXT NOT NULL`；
- `revoked_at TEXT NULL`；
- `created_at TEXT NOT NULL`。

为 session 的 `user_id`、`expires_at` 和活跃 token 查询建立必要索引。时间统一保存 UTC RFC 3339。ID 与 `jti` 使用注入的加密随机生成器；不得使用用户名、时间戳或自增值推导 token ID。

migration v2 落地后，补齐阶段一延后的真实 `schema_outdated` 集成测试：只应用 v1 的数据库必须分类为 outdated，运行 migrate 后升级到 v2 且保留已有 `seed_versions` 数据。migration 不写演示账号，避免 schema 与环境 seed 耦合。

auth seed 使用独立 group `auth_demo`、版本 `1`。development 的 `admin seed` 固定按 `runtime` → `auth_demo` 顺序执行；`admin reset` 删除并迁移后执行相同顺序，因此一并重建四个演示账号。任一 group 失败则整条命令返回失败且不得报告整体成功；每个 group 保持自身事务原子性，已提交的前序 group 不伪装成回滚。production 只运行 `runtime`，不写 `auth_demo` 版本或账号。

`auth_demo` 在 development 下创建或校验固定的 `viewer`、`operator`、`approver`、`admin` 用户；username 与 role 一一同名。seed version 小于 `1` 时在同一事务创建账号并写入版本；version 等于 `1` 时不得直接返回，仍需执行只读一致性校验，确认四个账号均存在、角色正确、未禁用且密码 hash 可验证当前 `DEMO_ACCOUNT_PASSWORD`。任何不一致均安全失败并要求显式 reset，不修改账号或 seed version；数据库 version 高于 `1` 时继续拒绝。重复 seed 不重复用户、不静默覆盖已变化的角色或密码。

production 首账号由 `cmd/admin bootstrap-admin` 创建。命令要求 schema 已迁移到最新、运行环境为 production、`users` 表当前没有任何记录，并使用单条条件写入 `INSERT ... SELECT ... WHERE NOT EXISTS (SELECT 1 FROM users)` 原子创建一个 role=`admin` 的账号；`RowsAffected` 不为 `1` 即拒绝。并发测试使用两个独立 `store.DB`/SQLite 连接打开同一个临时数据库，同时执行不同 username 的 bootstrap，证明跨连接竞争最多写入一条记录，而不只依赖单个 `sql.DB` 的 `MaxOpenConns=1` 串行化。成功输出只含非敏感账号摘要；用户表非空时无论 username 是否相同都拒绝，不更新、不重置密码。development 调用、重复调用、并发调用、缺失/非法输入或未迁移数据库均安全失败。后续账号与角色管理仍属于长期目标，不在本阶段扩展。

本阶段不自动删除过期或已撤销 session。development 可通过 reset 清库；production 数据会持续累积，因此这是已知运维限制，不得描述为生产完备。`expires_at`/`revoked_at` 索引为后续批量清理保留查询基础，定时或显式 purge 命令应在真实部署前单独设计并补事务、并发与审计测试。

## 5. 代码边界

```text
internal/
  auth/                 # claims、token 签发/验证、密码与授权策略
  store/                # users/session 持久化与 auth seed
    migrations/0002_auth.sql
  httpapi/              # auth middleware、context principal、三个 endpoint
  app/                  # 阶段二变更：配置、store、auth service、limiter 与 handler 装配
```

`internal/auth` 定义认证服务与 `Principal`，通过窄 store interface 访问用户和 session，不依赖 `http.Request`。`internal/httpapi` 负责 Bearer 语法、JSON、状态码和 context 注入，不直接执行 SQL 或解析密码哈希。角色比较使用显式 capability/allowlist，不依赖字符串排序或角色名称的字典序。

`internal/app` 负责从 API 配置构造 store、密码/JWT 服务、认证服务、limiter 和 HTTP dependencies，并按现有关闭顺序释放资源。HTTP 外层中间件顺序保持 request ID → access log → recovery；auth middleware 位于 recovery 之后，只包装需要认证的路由并在业务 handler 之前注入 principal。登录、health 和 readiness 不经过 auth middleware，业务 handler 不得绕过统一认证包装自行解析 Bearer token。

clock、ID 生成器和 limiter clock 由构造函数注入。生产默认实现使用 `time.Now` 与 `crypto/rand`；测试不得 sleep、修改进程全局时间或共享签名密钥。JWT 的 `iat` 必须满足 `iat <= now + 30s`；token 在 `now >= exp + 30s` 时失效，session 在 `now >= expires_at + 30s` 时失效，与 jwt/v5 的 `now < exp + leeway` 有效语义一致。登录创建 session 时，数据库 `expires_at` 必须与 JWT `exp` 完全相同。30 秒只容忍签发方与验证方的合理时钟偏差及临界请求，不改变 logout、禁用账号或角色变化的即时失效；相对 15 分钟 TTL 可接受，若未来统一可信时钟可重新收紧。

## 6. HTTP 契约收敛目标

### `POST /api/v1/auth/login`

- 请求：`{"username":"viewer","password":"..."}`，body 最大 4 KiB；
- 成功：`200`、`Content-Type: application/json`，返回 `{"accessToken":"...","tokenType":"Bearer","expiresAt":"...","user":{"id":"...","username":"viewer","role":"viewer"}}`；
- 认证失败：`401 AUTHENTICATION_FAILED`；
- 限流：`429 RATE_LIMITED`；
- 格式、类型、大小或字段错误：`400 INVALID_REQUEST` 或 `415 UNSUPPORTED_MEDIA_TYPE`。

### `GET /api/v1/auth/me`

- 要求唯一、格式正确的 `Authorization: Bearer <token>`；auth-scheme 按 HTTP 语义大小写不敏感，因此 `Bearer`、`bearer` 等等价，但 token 值保持原样且区分大小写；
- 成功：`200`、`Content-Type: application/json`，返回 `{"id":"...","username":"viewer","role":"viewer","tokenExpiresAt":"..."}`；
- 缺失、重复、错误 scheme、篡改、过期、错误 issuer/audience/alg、session 缺失/撤销/到期、`session.user_id` 与 `claims.sub` 不一致、账号禁用或角色变化：`401 UNAUTHENTICATED`。

### `POST /api/v1/auth/logout`

- 使用与 `/me` 相同认证链路；
- 成功撤销当前 session，返回 `204` 且无 body；
- 登录允许同一用户的既有 session 继续有效，logout 只撤销当前 token 对应 session；
- 两个已在撤销前通过认证 middleware 的并发 logout 均可幂等返回 `204`；不得因第二次条件更新未命中而返回 `500`；
- 撤销提交后新到达并使用旧 token 的请求在 middleware 返回 `401 UNAUTHENTICATED`。

三个 endpoint 的具体响应字段和错误码在实现测试中冻结后，从目标 API 迁入当前 API。所有响应继续复用阶段一 request ID 与错误 envelope；`401` 必须返回 `WWW-Authenticate: Bearer`，method 不匹配继续使用统一 `405` 与 `Allow`。

## 7. 实施批次

### 批次 A：配置、依赖与 migration

1. 为新增环境变量、长度边界和 admin/API 配置分离编写表驱动测试。
2. 核对并固定 bcrypt 与 JWT 模块版本，记录官方文档和安全选项结论。
3. 实现 migration v2、索引和版本升级测试，补真实 outdated 数据库集成用例。

聚焦验证：`go test ./internal/config ./internal/store ./internal/admin -count=1`。

### 批次 B：密码与身份原语

1. 实现角色、principal、密码 compare/hash 和 dummy compare。
2. 固定 UTF-8 byte 长度、bcrypt cost、username 规范化和加密随机 ID 原语。
3. 实现 user store 基础操作、development auth seed 与 production `bootstrap-admin`，覆盖幂等、篡改、环境、空表和并发边界。

聚焦验证：`go test ./internal/auth ./internal/store -count=1`。

### 批次 C：JWT、Session 与认证服务

1. 实现 claims 签发与严格解析，固定 HS256、issuer、audience、必需 claims 和 30 秒 leeway；parser 显式注入 clock并启用 issued-at 校验，自定义 validation 强制关键 claims 非空。
2. 实现 session store 操作及登录、认证、撤销服务事务边界。
3. 覆盖过期边界、未来 `iat`、篡改、错误算法/签发方/受众、重复 `jti`、subject/session 错配、撤销、禁用和角色变化。

聚焦验证：`go test ./internal/auth ./internal/store -count=1`。

### 批次 D：HTTP 与授权

1. 实现严格 login decoder、认证 middleware、principal context helper 和三个 endpoint。
2. 实现登录 limiter，注入 clock，固定全局 4096、每 IP 32 个 bucket、受限 bucket 保护和失败分类。
3. 实现角色策略 helper，并用仅存在于测试中的最小 handler 验证 viewer/operator/approver/admin 允许与拒绝矩阵；不向生产 mux 注册虚构业务路由。
4. 覆盖 method/path、body 边界、Bearer 语法、状态码/header、context 取消、store 关闭和内部错误。
5. 断言响应与结构化日志不泄露认证敏感值。

聚焦验证：`go test ./internal/httpapi ./internal/auth -count=1`。

执行备注：批次 D 先实现固定窗口、注入 clock、全局/每 IP 容量上限和过期清理，再严格按已冻结契约实现未封禁 bucket 淘汰；不得在实现中继续增加动态权重、概率淘汰或其他未规划启发式。限流测试先验证单 key 与窗口，再验证容量/淘汰，最后验证并发与 race，以控制调试范围。

### 批次 E：应用集成与文档收敛

1. 使用真实 HTTP 与临时 SQLite 完成 migrate → auth seed → login → me → logout 流程。
2. 交叉覆盖错误密码、未知/禁用账号、过期/篡改 token、subject/session 错配和撤销 session；角色越权由批次 D 的测试专用 HTTP handler 提供 transport/policy 集成证据，真实业务路由越权留到阶段三。
3. 更新当前/目标 HTTP API、配置说明、领域模型、验证矩阵、路线图、README 和 CHANGELOG。
4. 将认证门禁改为 `enabled: yes`，记录实际包和命令；不得提前启用订单或页面门禁。

最终验证：

```sh
go test ./... -count=1
go vet ./...
go test -race ./...
```

## 8. 完成标准

- 四个 development 演示账号可通过同一登录入口取得各自角色的短期 JWT；
- 全新 production 数据库可在 migrate 后通过一次性 CLI 安全创建首个 admin，重复或非空库调用被拒绝；
- JWT 必须同时通过签名、固定算法、claims、session、用户状态和当前角色检查；
- logout、账号禁用、角色变化和 session 到期可使已签发 token 立即或按时间规则失效；
- 认证失败不暴露账号是否存在，限流与严格输入边界生效；
- migration v2 可从 v1 安全升级，真实 outdated readiness 测试补齐；
- 密码、哈希、token、密钥和 session token ID 不出现在响应或日志；
- 真实 HTTP + 临时 SQLite 集成覆盖验证矩阵要求，所有质量门禁通过；
- 三个 endpoint 已迁入当前 API，目标草案、路线图和 CHANGELOG 与实现一致。

## 9. 非目标

- refresh token、Cookie session、OIDC、MFA、密码找回或密钥轮换；
- 账号创建、禁用、改密、角色修改等 admin HTTP API；
- 分布式或持久化限流、可信代理链和多实例 session cache；
- 过期/已撤销 session 的后台或定时清理；development 依赖 reset，production 持续增长是已记录的运维限制；
- 浏览器 CORS；延后到阶段三首个 Schema-UI 前端联调批次冻结可信源与预检策略；
- Unicode username NFKC/同形字符归一化；首版只做 trim + lowercase，因此视觉相似用户名可能并存；
- 订单、退款、附件、看板或页面配置 endpoint；
- 为后续业务 endpoint 预先冻结资源级授权规则或错误码。

## 10. 风险与控制

| 风险 | 控制 |
|---|---|
| JWT algorithm confusion 或 claims 缺失 | parser 固定 HS256、issuer、audience及必需 claims，并用错误算法 token 测试 |
| 用户名枚举 | 统一响应、dummy hash compare、相同限流路径和日志脱敏 |
| 已撤销 token 继续可用 | 每次认证查询 session，不以 JWT 签名成功代替 session 检查 |
| 账号禁用或角色变化不即时生效 | 每次认证读取当前用户并比较 token role |
| SQLite 并发登录/登出竞争 | 唯一 `jti`、条件撤销 SQL、事务测试和 race detector |
| limiter 内存增长或单 IP 容量攻击 | 全局 4096、每 IP 32 个 bucket；只淘汰未封禁最旧项并保护活跃限制；执行跨 IP 高基数测试 |
| 演示密码进入仓库或日志 | 只从环境读取，hash 后存储，测试扫描响应与日志 |
| migration 混入环境数据 | v2 只建 schema，演示账号由独立 development seed group 管理 |
| bcrypt 输入边界或参数不当 | 开工时核对官方文档，固定 cost，测试 72-byte 边界与 hash 升级判断 |
| JWT 密钥导致 admin 命令不可用 | auth-free admin 命令不加载密钥，只有 API/auth 装配要求密钥 |
