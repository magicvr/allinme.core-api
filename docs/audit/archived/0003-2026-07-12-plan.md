# 阶段三：订单查询与履约开发计划

配套路线：[实施路线图](../../06-implementation-roadmap.md)。本阶段只实现订单查询、订单写入和履约状态变化；退款、附件、看板、页面配置和账号管理不进入本轮实现。

## 1. 目标与完成边界

在阶段二认证主体之上，建立可持久化的订单核心，使四类角色能够通过真实 HTTP API 完成：

- `viewer`、`operator`、`approver`、`admin` 查看订单列表与详情；
- `operator`/`admin` 创建草稿、编辑草稿并推进确认、履约、发货、完成或取消；
- 列表支持关键词、订单状态、支付状态、创建日期、allowlist 排序和服务端分页；
- 所有金额由订单项以整数最小货币单位计算，服务端不接受客户端总额作为事实；
- 创建型请求支持幂等键，状态变化和编辑使用 `version` 乐观锁；
- 响应提供基于当前主体和资源状态计算的 `canEdit`、`canAdvance`、`canCancel` 展示字段；阶段三尚无退款 API，因此 `canRequestRefund`、`canApproveRefund` 恒为 `false`，所有 capability 字段均不替代后端授权；
- development seed 覆盖订单状态和支付状态，为阶段四退款提供前置数据。

完成本阶段不代表页面已实现。Schema-UI 页面配置属于阶段六；本阶段只提供页面所需的稳定 API 和集成证据。订单 endpoint 从目标 API 迁入当前 API 时，阶段三只要求真实 HTTP、业务、权限、并发和稳定契约测试；对应 Schema-UI datasource/Action 页面场景只在页面实际引用该 endpoint 时由阶段六门禁覆盖，阶段三最小跨源 CORS smoke 不得替代这些页面场景。

## 2. 领域决策摘要

[订单运营领域模型](../../05-domain-model.md)是状态机、角色能力、金额范围和业务不变量的唯一事实源。本节仅记录阶段三执行摘要和数据库/service 落地决策；若与领域模型冲突，必须先修改领域模型，再同步本计划和测试。

订单履约状态严格为：`DRAFT -> CONFIRMED -> FULFILLING -> SHIPPED -> COMPLETED`；`DRAFT`、`CONFIRMED`、`FULFILLING` 可取消，终态不可取消。创建/编辑只允许 `operator` 和 `admin`，查看允许全部角色。创建后的 `version` 初值固定为 `1`，编辑或 Action 成功后原子 `+1`。`createdAt` 创建后不可变；成功 edit/Action 使用注入 clock 设置新的 `updatedAt`，失败请求和幂等重放不修改任一时间字段。

编辑和 Action 采用“事务内读取当前资源 → 校验 version 与状态 → 同时匹配 ID、源状态和 version 的条件更新”。首次读取或条件更新后的重新读取固定按存在性 → version → 状态的顺序分类：资源不存在为 `NOT_FOUND`，资源存在但请求 version 不同为 `VERSION_CONFLICT`，version 相同但源状态不允许为 `STATE_CONFLICT`；version 与状态同时不匹配时返回 `VERSION_CONFLICT`。并发导致条件更新未命中时不得一律映射为版本冲突，也不得向客户端暴露 SQL 行数或底层 SQLite 错误。

支付状态的数据库枚举固定允许 `UNPAID`、`PAID`、`PARTIALLY_REFUNDED`、`REFUNDED`，避免阶段四修改枚举约束；阶段三创建写路径只能产生 `UNPAID`，阶段三 seed 只写 `UNPAID` 或 `PAID`，任何 HTTP 写路径均不得写入后两个退款状态。订单币种首版只允许 `CNY`。金额使用 `int64` 表示分，`unitPrice` 固定为 `1..9_999_999_999`，订单 `totalAmount` 固定为 `1..9_999_999_999`，数量固定为 `1..10_000`；每次乘法和加法先执行 checked arithmetic，再校验总额上限，数据库同时以 CHECK 保护范围。

capability 精确语义固定如下：`canEdit` 仅对 `operator`/`admin` 的 `DRAFT` 为 `true`；`canAdvance` 仅对 `operator`/`admin` 且存在唯一下一步履约 Action 时为 `true`，即 `DRAFT`、`CONFIRMED`、`FULFILLING`、`SHIPPED`，`COMPLETED`/`CANCELLED` 和 viewer/approver 恒为 `false`；`canCancel` 独立计算，仅对 `operator`/`admin` 的 `DRAFT`、`CONFIRMED`、`FULFILLING` 为 `true`；阶段三 `canRequestRefund`、`canApproveRefund` 恒为 `false`。

订单和明细 ID 分别固定为 `ord_`/`itm_` 加 32 个小写十六进制字符，来自 128-bit `crypto/rand`，不得由时间、客户、数据库行号或数量推导。development seed 也使用相同格式的固定 hex ID；固定性只用于可重复 seed，不放宽 HTTP path 校验。时间由注入 clock 生成，repository 写入前调用 `UTC()` 并固定以 Go `time.RFC3339` 的 `Z` 形式保存和返回；SQLite CHECK 只验证文本形状与 `Z` 后缀，真实可解析性和 UTC 规范化由 repository 测试保证。

数据库最终使用 `orders`、`order_items`、`idempotency_keys` 三类业务表。M1 的 migration v3 只创建 `orders`/`order_items` 及查询索引，M2 的 migration v4 再创建 `idempotency_keys` 及其索引；验收复核追加 additive migration v5，为幂等记录增加独立 `snapshot_digest`，并在同一迁移事务中为既有 v4 snapshot JSON 回填 SHA-256，不改变 snapshot v1 DTO 或 request digest 语义。不得在只读里程碑提前引入未使用的幂等 schema。`order_items` 保存从 `0` 开始、订单内唯一的 `position`，详情固定按 `position ASC` 返回；外键使用 `order_id REFERENCES orders(id) ON DELETE CASCADE`。阶段三不提供删单 API，幂等记录不随订单单独清理，development reset 通过重建数据库统一清除。订单 service 通过窄 repository 接口协调业务事务语义、权限、状态、version、金额和幂等；store 提供聚合级原子 repository 方法或事务执行器，repository 不决定状态转换，handler 不直接执行 SQL 或拼装领域规则。

订单 service 直接接收 `auth.Principal` 并使用 `auth.RoleAllowed(principal.Role, auth.RoleOperator, auth.RoleAdmin)`；不得在 `internal/order` 重新声明 role、复制角色字符串或建立隐式角色排序。`admin` 是订单写能力的显式允许角色，与领域模型和目标 API 保持一致。

## 3. 已冻结的 HTTP 与幂等契约

本节和 §4 是阶段三 active 期间详细 HTTP/CORS 契约的唯一事实源；[目标 HTTP API](../../03-http-api-target.md)只维护 endpoint 范围与跨阶段边界。实现完成后，对应契约迁入[当前 HTTP API](../../03-http-api.md)，本节改为实现摘要而不继续维护第二份字段清单。

| 项 | 阶段三冻结值 |
|---|---|
| 创建幂等 header | `Idempotency-Key`；必填，长度 `1..128` bytes，正则 `[A-Za-z0-9][A-Za-z0-9._:-]{0,127}`；缺失或非法返回 `400 INVALID_REQUEST`，`error.details[0].field` 为 `Idempotency-Key` |
| 幂等适用范围 | 仅 `POST /api/v1/orders`；`PATCH` 和五个 Action 即使携带 `Idempotency-Key` 也明确忽略该 header，只使用 `version`，不得返回 400 或写幂等记录 |
| 幂等作用域 | 数据库唯一键固定为 `principal user ID + method + route/operation + Idempotency-Key`。阶段三只有 `POST /api/v1/orders` 一个幂等 endpoint；相同主体、method、route/key 的不同 normalized request digest 返回 `409 IDEMPOTENCY_CONFLICT`。阶段四新增幂等 endpoint 时可复用同一 key 而不与订单创建冲突；method/route 仍作为记录完整性元数据核对，错配视为损坏记录并返回 `500 INTERNAL_ERROR` |
| 幂等存储结果 | 保存 `snapshot_version=1` 与独立 snapshot v1 JSON；它表示首次创建成功时 version=1 的订单结果，不查询或拼接当前订单。首次与重放返回语义等价的同一快照，即使订单后来已编辑或推进也不改变；未知版本、损坏 JSON 或元数据错配返回 `500 INTERNAL_ERROR`，绝不重新创建 |
| snapshot v1 DTO | 顶层仅含 `order`；order 固定包含字符串 `id/customerName/status/paymentStatus/currency/createdAt/updatedAt`、int64 `totalAmount/version`、数组 `items`；item 固定包含字符串 `id/sku/name` 与 int64 `quantity/unitPrice`。items 按 position 排序但不保存 position；不保存 capability、request ID、HTTP status/header 或 error envelope |
| snapshot 重放映射 | service 从 snapshot v1 恢复创建结果，再按当前已认证 principal 与快照中的 `DRAFT` 状态重新计算 capability；由于 create 只允许 operator/admin，两者 capability 相同。不得查询当前 orders/order_items 补字段，响应中的订单事实字段必须来自不可变快照 |
| snapshot 元数据校验 | 幂等记录独立列固定保存并在读取时核对 principal user ID、key、method、route/operation、request digest、snapshot digest、order ID、snapshot version；`request_digest` 只覆盖 normalized request facts，`snapshot_digest` 覆盖完整 snapshot JSON（包括服务端生成的 order/item ID）。阶段三记录必须为 method=`POST`、route=`/api/v1/orders`。snapshot 内 order ID 必须与列值相同，status/paymentStatus/version 必须为 `DRAFT`/`UNPAID`/`1`，`createdAt == updatedAt`，时间/金额/枚举/items 必须通过 snapshot v1 解码校验 |
| 请求摘要 | 严格解码后先将 customerName/SKU/item name trim、校验 currency/数值并形成 normalized typed command，再按固定字段顺序编码 canonical JSON；保留明细顺序，忽略原始 JSON 空白、对象字段顺序和字符串边缘空白，因此 `"Alice"` 与 `" Alice "` 使用同一摘要；摘要为 SHA-256，并包含固定操作标识 `POST /api/v1/orders` |
| Content-Type | create/edit/Action 必须提供可由 `mime.ParseMediaType` 解析且主类型为 `application/json` 的 Content-Type，可携带合法参数；缺失、其他媒体类型或非法值返回 `415 UNSUPPORTED_MEDIA_TYPE` |
| 当前 body 预算 | create/edit 当前预算为 `64 KiB`，Action 为 `1 KiB`；超限返回 `400 INVALID_REQUEST`，未知字段和多余 JSON 值同样拒绝。预算可在不改变错误语义和安全上限原则时通过 plan 决策调整，不视为长期兼容承诺 |
| JSON 命名与 create body | 所有 JSON 字段使用 camelCase；create 固定为 `{"customerName":"...","currency":"CNY","items":[{"sku":"...","name":"...","quantity":1,"unitPrice":100}]}`，不接受客户端 ID、total、status、paymentStatus 或 version |
| edit / Action body | edit 为与 create 相同三个事实字段外加整数 `version`；五个 Action body 仅允许 `{"version":n}`，不接受目标状态或其他字段 |
| 字符串与明细 | `customerName` trim 后为 `1..120` UTF-8 bytes；`sku` trim 后为 `1..64` bytes；item `name` trim 后为 `1..160` bytes；保存 trim 后值；`items` 数量为 `1..100`，按请求数组顺序分配 `position` |
| currency | create/edit 必填且只能精确为 `CNY`，不提供缺省值；缺失或其他值为 `422 VALIDATION_FAILED` |
| version | 创建为 `1`；每次成功 edit/Action 原子 `+1`；edit/Action 缺失、null、字符串、小数或超出 int64 属于 JSON 结构错误并返回 `400 INVALID_REQUEST`，解析成功但 `version <= 0` 返回 `422 VALIDATION_FAILED`，正整数但与资源不一致返回 `409 VERSION_CONFLICT` |
| 分页 | query 为 `page`、`pageSize`；默认 `1`/`20`，二者只接受普通十进制正整数并解析为 int64，`page >= 1`，`pageSize=1..100`。offset 使用 checked arithmetic 计算 `(page-1)*pageSize`；page 超出 int64 或乘法溢出返回 `400 INVALID_REQUEST`，`error.details[0]` 固定为 `{"field":"page","message":"page is too large"}`。只要 offset 可表示，任意大 page 都合法并安全返回空 `items`、真实 `total` 和请求中的 page/pageSize，不得产生负 offset、驱动错误或 500 |
| 筛选 query | `q`、`status`、`paymentStatus`、`createdFrom`、`createdTo`；必须对 `url.ParseQuery(RawQuery)` 的错误显式返回 `400 INVALID_REQUEST`，不得静默忽略畸形百分号编码或非编码分号；`q` trim 后最多 200 UTF-8 bytes；日期接受 Go `time.RFC3339` 的 `Z` 或合法 offset，解析后规范化为 UTC，范围为闭区间；重复参数拒绝 |
| 排序 | query 为 `sort`、`order`；allowlist 为 `createdAt`、`updatedAt`、`totalAmount`、`customerName`、`status`，方向仅 `asc`/`desc`，默认 `createdAt desc`；所有排序固定追加 `id` 同方向作为唯一二级排序，即 `ORDER BY <allowlisted column> <direction>, id <direction>` |
| 列表 envelope | `{"items":[],"total":0,"page":1,"pageSize":20}`；`total` 是应用全部筛选后的总数；COUNT、当前页查询和非空页的一次批量明细完整性查询必须在同一只读事务中执行，保证来自同一 SQLite snapshot；列表不返回 items，但必须拒绝无明细、position 不连续、字段非法、金额溢出或 totalAmount 与明细合计不一致的损坏聚合 |
| Order 响应 DTO | 固定包含 `id/customerName/status/paymentStatus/currency/totalAmount/version/createdAt/updatedAt/canEdit/canAdvance/canCancel/canRequestRefund/canApproveRefund`；所有字段 camelCase，时间为 UTC RFC 3339 |
| Item 响应 DTO | 详情/create/edit/Action 中每项固定包含 `id/sku/name/quantity/unitPrice`；`position` 仅用于稳定排序，不进入 JSON；列表项不含 `items` 字段 |
| 成功状态 | create 首次为 `201` + 首次创建的完整订单结果，重放为 `201` + 语义等价的 snapshot v1；edit 与五个 Action 均为 `200` + 本次成功写入后的当前订单。列表项不含 `items`，详情/create/edit/Action 响应包含按 `position` 排序的 `items` |
| orderId path | 仅接受 `ord_` + 32 位小写 hex；格式非法与格式合法但不存在均返回 `404 NOT_FOUND`，避免暴露 ID 校验细节 |
| 详情附件 | 阶段三不返回 `attachments` 字段，也不返回附件 ID、URL 或附件 capability；阶段五实现时新增并冻结该字段 |
| 错误 details | `error.details` 为可选数组，元素固定为 `{"field":"items[0].quantity","message":"must be between 1 and 10000"}`；无字段错误时省略整个 `details`，不返回 `[]`/`null`；JSON 字段用 camelCase path，query 用参数名如 `pageSize`，header 用规范 header 名 `Idempotency-Key`；元素按校验顺序稳定返回，message 为安全英文文本 |
| 错误机器码 | `INVALID_REQUEST`(400)、`UNAUTHENTICATED`(401)、`FORBIDDEN`(403)、`CORS_ORIGIN_DENIED`(403)、`CORS_PREFLIGHT_DENIED`(403)、`NOT_FOUND`(404)、`METHOD_NOT_ALLOWED`(405)、`VERSION_CONFLICT`(409)、`STATE_CONFLICT`(409)、`IDEMPOTENCY_CONFLICT`(409)、`UNSUPPORTED_MEDIA_TYPE`(415)、`VALIDATION_FAILED`(422)、`INTERNAL_ERROR`(500)、`SERVICE_UNAVAILABLE`(503) |
| GET endpoint 的 HEAD | 阶段三不支持 HEAD；订单列表/详情 GET 路由必须显式返回 `405 METHOD_NOT_ALLOWED`，准确提供该 route 的 `Allow`，不得因 `http.ServeMux` 的 GET/HEAD 匹配行为意外返回成功 |

`INVALID_REQUEST` 专用于 transport/结构错误：JSON 语法、类型、未知字段、多余值、body 超限、缺失/非法幂等 header，以及非法 query 结构、枚举或范围。`UNSUPPORTED_MEDIA_TYPE` 专用于缺失、非法或非 JSON Content-Type。`VALIDATION_FAILED` 专用于 typed command 的业务字段语义：trim 后空字符串、字符串/明细/数量/金额边界、缺失或非 `CNY` 币种。认证、授权、存在性和冲突分别使用冻结的 401/403/404/409 语义；字段级错误 detail 使用请求中的 camelCase 字段路径。

阶段三扩展现有公共错误 envelope：`code`、`message`、`requestId` 保持不变，只在字段可定位的 `INVALID_REQUEST`/`VALIDATION_FAILED` 中可选增加 `details`。认证、授权、资源、冲突、内部和服务不可用错误不返回 `details`；阶段一/二客户端忽略新增可选字段即可保持兼容。

订单 HTTP 请求固定按以下顺序短路，组合错误不得改变优先级：已知路由/path pattern 与 method → CORS actual/preflight 检查 → authentication → endpoint role authorization → Content-Type → create 的 `Idempotency-Key` / query / orderId path 结构 → JSON 结构解码 → typed command validation → repository 资源存在性 → version → state。CORS middleware 使用显式 route allowlist 先判定已知 path/method，因此 unknown path 保持 404；对已知实际请求，非法 Origin 先于认证拒绝。只有通过前一层才执行后一层；例如非法 Origin + 缺失 token 返回 CORS 403，允许 Origin 下 viewer 的错误 Content-Type 写请求返回业务 `403`，operator 的非法 Content-Type + 损坏 JSON 返回 `415`，operator 的非法 orderId + 损坏 JSON 返回 `404`。

关键词 `q` 固定在规范化查询中匹配订单 ID、客户名、SKU 和明细名；trim 后空字符串等同未提供，`%`、`_` 和转义字符按字面量处理。阶段三冻结为 SQLite 可可靠支持的 ASCII 大小写不敏感：ASCII 字母先统一折叠后比较，中文等无大小写字符按原值比较；不声称完整 Unicode case folding，也不依赖默认 `NOCASE` 处理非 ASCII。`createdFrom > createdTo` 返回 `400 INVALID_REQUEST`。同一 query 参数出现多次一律拒绝，不采用首值或末值。

## 4. CORS 契约

阶段二延期的浏览器 CORS 在阶段三 M3 落地。API 配置新增可选 `CORS_ALLOWED_ORIGIN`：未设置时保持当前无 CORS 行为；设置时必须是单个绝对 `http`/`https` origin，仅含 scheme、host 和可选 port，不允许 `*`、userinfo、path、query 或 fragment。development/production 均不提供内置默认值，真实 Schema-UI 联调必须显式配置可信 origin。

- 无 `Origin` 的同源/非浏览器请求按原路由处理；`Origin` 与 `Access-Control-Request-Method` 都是严格单值 header，重复值必须拒绝；匹配的实际跨源请求返回精确 `Access-Control-Allow-Origin` 和 `Vary: Origin`，不启用 credentials。
- 预检 `OPTIONS` 位于 authentication 之前，仅对匹配 origin、目标 method 和请求 header 成功返回 `204`；允许 method 为 `GET, POST, PATCH, OPTIONS`，允许 header 为 `Authorization, Content-Type, Idempotency-Key, X-Request-ID`，`Access-Control-Max-Age: 600`。
- 预检成功和拒绝结果都返回与缓存判断因素一致的 `Vary: Origin, Access-Control-Request-Method, Access-Control-Request-Headers`；实现必须追加或合并 `Vary`，不得覆盖其他 middleware 已写入的值。实际响应至少保留 `Vary: Origin`。
- 实际请求与预检的 origin 不匹配、origin 非法、请求 method/header 不在 allowlist 时返回 `403 CORS_ORIGIN_DENIED` 或 `403 CORS_PREFLIGHT_DENIED`，复用 request ID error envelope；不返回 `Access-Control-Allow-Origin`。
- 响应公开 `X-Request-ID`，即 `Access-Control-Expose-Headers: X-Request-ID`。CORS 只控制浏览器 transport，不替代 authentication、authorization 或业务校验。

完整 middleware 顺序固定为 request ID → access log → recovery → CORS → authentication → authorization → route handler，与阶段二已冻结的前三层保持一致。CORS 拒绝和预检均已处于 request ID、access log、recovery 内，因此返回 `X-Request-ID` 并进入访问日志。无 `Origin` 请求完全跳过 CORS；非 OPTIONS 的非法/不匹配 Origin 在 authentication 前拒绝。`OPTIONS` 只对显式 CORS route allowlist 中的已知 API path 处理预检，未知 path 返回普通 `404` 且不返回 CORS allow header。

配置装配固定为 `config.LoadAPI` 校验 `CORS_ALLOWED_ORIGIN` → `config.APIConfig.CORSAllowedOrigin` → `app.NewAuthenticatedAPIWithDependencies` → `httpapi.Dependencies.CORSAllowedOrigin`。非法 origin 必须使 `config.LoadAPI` 失败，进程不得继续到 app 构造、监听端口、创建/打开数据库或执行 migration。配置测试覆盖 development/production 的未设置、合法 http/https、通配符、多个 origin、userinfo、path/query/fragment；未设置始终合法且表示禁用，不因环境隐式启用。

应用装配边界固定为：只有 `app.NewAuthenticatedAPI`/`NewAuthenticatedAPIWithDependencies` 打开共享 `store.DB`、创建 auth service、order repository 和 order service，并将非 nil `OrderService` 与 `CORSAllowedOrigin` 放入 `httpapi.Dependencies` 后注册订单路由；任一步失败都使构造器返回错误且不启动 HTTP server。`app.NewAPI` 继续只提供健康检查、readiness 和 fallback，不打开业务数据库、不注册认证或订单路由；`OrderService == nil` 时 `httpapi.NewHandler` 不注册订单路由，不得留下会 panic 的半注册 handler。测试必须证明基础 API 的订单 path 为 404，认证 API 才注册订单路由且仍经过 auth middleware。

生命周期顺序固定为：先停止接收请求并在配置期限内完成 `http.Server.Shutdown`；超时后调用 `http.Server.Close` 强制断开，禁止使用无期限 context 再次等待；随后关闭 readiness probe，使 order/auth service 不再可达，再关闭二者共享且由 app 拥有的 `store.DB`，最后释放进程锁。导出的 `API.Close` 与 `Run` 必须维护同一顺序。order/auth service 不拥有数据库且阶段三不单独引入无意义的 `Close`；若未来增加自有可关闭资源，必须在共享 DB 之前关闭并增加顺序测试。`httpapi.NewHandler` 只注册 route/middleware，不创建数据库或 service；订单 handler 通过 `PrincipalFromContext` 获取主体，order service 接收 Principal 并执行业务授权。共享 `routeMetadata` allowlist 同时驱动 known path、mux 注册、支持 method、OPTIONS 预检 method/header 和 method mismatch 的 `Allow`，测试证明三者不会漂移。

## 6. Seed 基线

`order_demo=1` 使用符合生产格式的固定 hex seed ID，并冻结以下最小组合；固定 ID 只用于 development seed，展示含义由 `customerName` 表达，生产创建仍使用随机 ID：

| 固定订单 ID | 履约状态 | 支付状态 | 后续用途 |
|---|---|---|---|
| `ord_00000000000000000000000000000001` | `DRAFT` | `UNPAID` | 编辑、确认、取消 |
| `ord_00000000000000000000000000000002` | `CONFIRMED` | `UNPAID` | 开始履约、取消 |
| `ord_00000000000000000000000000000003` | `FULFILLING` | `PAID` | 发货、取消、阶段四退款前置 |
| `ord_00000000000000000000000000000004` | `SHIPPED` | `PAID` | 完成、阶段四退款前置 |
| `ord_00000000000000000000000000000005` | `COMPLETED` | `PAID` | 阶段四部分/全额退款前置 |
| `ord_00000000000000000000000000000006` | `CANCELLED` | `UNPAID` | 终态冲突 |

上述组合只约束阶段三 HTTP 写路径和 `order_demo=1` seed，不作为 orders 表的交叉 CHECK，也不使 repository 读取其他“两个枚举各自合法”的组合失败。数据库只分别约束订单状态与支付状态枚举；因此 `DRAFT + PAID` 不视为 schema 损坏，但阶段三 HTTP/seed 不产生它。履约 Action 不改变支付状态。阶段四扩展写路径或 seed 组合时必须先更新领域模型与本契约。

六个 seed 订单的 `created_at` 固定为 `2026-01-01T00:00:00Z` 至 `2026-01-01T05:00:00Z`，按表中顺序每条递增一小时；每条 `updated_at == created_at`。`SeedOrderDemo(ctx, appliedAt)` 接收注入时间，仅用于新写入的 `seed_versions.applied_at`，生产默认传 `time.Now().UTC()`，测试传固定值；已是 version 1 的只读一致性校验不更新时间。排序与场景断言只依赖订单固定时间，不依赖 seed 元数据时间。

admin 的 development `seedDemo`/`resetDemo` 固定按 `runtime → auth_demo → order_demo` 调用；`SeedOrderDemo` 只接收 `appliedAt`，admin production path 传 `time.Now().UTC()`，测试直接传固定时间。每个 group 的结果和 seed_versions 行都可断言顺序；重复 seed 只读校验且不更新时间。

## 7. 代码边界

```text
internal/
  order/                 # 状态机、金额、capability、OrderService 与窄 repository 接口
  store/                 # migration v3、订单/明细/幂等 repository、订单 seed
  httpapi/               # orders 路由、严格 JSON/query 解码、认证和错误映射
  app/                   # 订单 service 与 repository 的依赖装配
```

包名固定为单数 `internal/order`，与 `internal/auth` 的领域包风格一致。`internal/order` 不依赖 HTTP 和 SQL，并定义 `OrderService` 及其所需的窄 repository 接口；`internal/store` 只实现参数化查询、事务和数据库约束；`internal/httpapi` 只做 transport、认证主体获取和错误映射，不直接拼 SQL、不直接执行状态转换。`OrderService` 是状态、权限、version、幂等和金额不变量的唯一协调者。

构造器边界同样属于代码架构：`app.NewAPI` 是不带业务认证依赖的基础模式，不装配 order package；`app.NewAuthenticatedAPIWithDependencies` 是订单 API 的唯一应用入口，必须先成功创建共享 DB、auth service、order repository/service，再构造 handler。`httpapi.NewHandler` 以非 nil `OrderService` 作为订单 route registration 的前提，不能注册后在请求期才因 nil 依赖返回 panic 或临时 500。

repository 接口推荐使用以下业务类型形态，不暴露 `*sql.DB`、`*sql.Tx` 或 SQL 字符串；方法名和参数组织允许等价调整，但必须保留窄接口和 service 不接触 `database/sql` 的边界：

```go
type OrderRepository interface {
  ListOrders(context.Context, ListQuery) (Page, error)
  GetOrder(context.Context, string) (Order, error)
  CreateOrder(context.Context, CreatePersistence) (CreateResult, error)
  UpdateDraft(context.Context, UpdateDraftPersistence) (Order, error)
  TransitionOrder(context.Context, TransitionPersistence) (Order, error)
  ReplayCreate(context.Context, ReplayKey) (CreateSnapshot, error)
}
```

用例层拥有业务事务语义，store 拥有 `database/sql` 事务对象和提交/回滚实现。阶段三的单订单操作允许由 `ListOrders`、`CreateOrder`、`UpdateDraft`、`TransitionOrder` 等聚合级 repository 方法内部完成原子事务；这只是阶段三实现选择，不是后续退款、附件等跨 repository 操作的不可变边界。后续跨聚合原子操作应由用例层通过 store 提供的事务执行器协调。`OrderService` 编排权限、金额、version、状态和幂等 command，但不接触 `database/sql`。列表必须在显式只读事务中执行一次 COUNT、一次 page query，并在当前页非空时执行一次批量明细完整性查询；事务 BEGIN/COMMIT 不计入上限，禁止逐订单 item 查询或逐行事务。测试通过 store 内部 query observer 统计调用，对空页断言 2 次，对 1 条和数百条 fixture 的非空页断言固定 3 次；该测试 seam 不扩大生产 repository 接口。筛选 query 可使用参数化 `EXISTS` 访问 `order_items` 以支持 `q` 对 SKU/name 的筛选；批量查询只校验当前页聚合，不把 items 暴露到列表 DTO。列表 capability 只由 principal 与订单 status 计算；详情返回有序 items。

冻结的是外部契约和可观察边界：endpoint、角色矩阵、状态转换、金额范围、DTO、错误码、幂等语义、分页排序、CORS header/短路顺序、事务 snapshot 一致性、查询有界性和取消/日志结果。以下属于推荐实现而非不可替换 API：repository 方法名、`applyMigration` helper 名称、具体 executor/barrier seam、测试函数名，以及使用 `http.ErrAbortHandler` 的具体连接中止机制。实现可以采用等价方案，但必须提供相同的事务一致性、查询上限、连接中止、错误分类和资源清理证据。

变更分层固定如下：

| 层级 | 内容 | 调整规则 |
|---|---|---|
| 外部硬契约 | HTTP endpoint、DTO、角色/状态、错误码与 header、幂等/version、分页排序、CORS 可观察行为；`Retry-After: 1` 属于客户端可见 header | 实现前先修改 plan §2-§4、关联 API/领域文档和契约测试 |
| 架构硬边界 | 用例层拥有业务事务语义、store 拥有 SQL 事务实现、service/repository/handler 职责、无逐行查询、auth 前提、错误分类和资源生命周期 | 阶段三可使用聚合级原子 repository 方法；跨聚合事务改用 store 事务执行器时须保留窄接口并记录等价证据 |
| 推荐验证方式 | 测试函数名、helper 名、barrier/executor 形式、`EXPLAIN QUERY PLAN`、fixture 数量、测试 busy timeout；`SetMaxOpenConns(1)` 仅描述当前基线 | 可用等价证据替换，无需修改外部契约；在 checklist 证据栏记录实际方式即可 |

错误分类矩阵：

| 错误 | 拥有/产生层 | service 处理 | HTTP 映射 |
|---|---|---|---|
| `context.Canceled` / `context.DeadlineExceeded` | 任意 I/O 层 | 原样保留，保持 `errors.Is` | 若尚未提交响应则通过 `http.ErrAbortHandler` 或等价机制中止连接，不映射 503，不写业务 envelope；access log 记为 canceled outcome |
| `NotFound` | store/order repository | 原样保留 | `404 NOT_FOUND` |
| `VersionConflict` | order service/repository 条件更新分类 | 原样保留 | `409 VERSION_CONFLICT` |
| `StateConflict` | order service | 原样保留 | `409 STATE_CONFLICT` |
| `IdempotencyConflict` | order service 比较 digest | 原样保留 | `409 IDEMPOTENCY_CONFLICT` |
| `Unavailable` | store 仅从 SQLite BUSY/LOCKED 转换 | 原样保留 | `503 SERVICE_UNAVAILABLE` + `Retry-After: 1` |
| `Validation` | order typed command 校验 | 携带安全 field details | `422 VALIDATION_FAILED` |
| 损坏行/snapshot、关闭 DB、其他 SQL/scan | store | 包装为 typed `Internal`，保留 cause 供日志但不对外 | `500 INTERNAL_ERROR` |

错误类型支持 `errors.Is`/`errors.As`；store 通过驱动错误码分类 BUSY/LOCKED，不匹配错误字符串。HTTP 只对 exported/typed 分类做映射，不读取错误文本决定状态码。

## 8. 端到端里程碑

阶段三只设三个对外里程碑。每个里程碑必须形成可启动、可通过真实登录验证、可独立合并和可独立关闭的纵向切片；包级提交可以在里程碑内部拆分，但不能用“尚未装配”作为对外里程碑完成状态。

schema migration 只前进，不提供 down migration。M1/M2 的 v3/v4 均必须是 additive migration；“回退里程碑”固定表示关闭对应路由或回退功能代码，同时保留已迁移 schema 和已有数据。兼容基线按 schema 版本分别交付和验证：M1-A binary 内含 v3 migration、可在 v3 数据库启动但不注册订单路由；M2-B0 binary 内含 v4 migration、可在 v4 数据库启动并保留 M1 查询，但不注册任何订单写路由。只有对应 schema-only binary 已部署、migration/readiness 与旧功能 smoke 已通过，才允许启用随后里程碑的外部路由。原始只支持 v2 的阶段二 binary 不是 v3 数据库的合法回退目标，M1-B/M2-A 的 v3 binary 也不是 v4 数据库的合法回退目标。migration 自身的“回滚”仅指单个 migration 执行失败时事务回滚、`user_version` 不前进，不表示 v3/v4 降级。若无法提供对应版本的兼容基线 binary，则执行该次 migration 前必须停写并备份，回退只能通过恢复整库完成，并明确丢弃 migration 后写入的数据。

P0-3a 在开工前按以下 schema-only 发布 Evidence 模板冻结 v3、v4 的目标环境、产物保存规则、路由关闭机制、配置来源、数据库停写/备份/恢复步骤、回退触发条件和数据损失口径；此时不要求填写尚未实现的实际 revision、产物 hash 或演练结果。M1-1/M2-3 在对应 schema-only 代码完成后，必须使用固定产物填写实际值和原始命令输出位置，并在启用后续外部路由前完成迁移、smoke 和回退演练证据：

| Evidence 字段 | 必须记录的实际信息 |
|---|---|
| schema 与产物 | 目标 schema v3/v4；Git revision；API/admin binary、镜像或发布产物的标识和 SHA-256；构建命令；产物保留位置 |
| 路由关闭机制 | schema-only 产物如何保证关闭本里程碑路由；API 启动命令；`APP_ENV`、`PORT`、`DATA_DIR` 等非秘密配置来源；`JWT_SIGNING_KEY` 等秘密只记录来源名称，不记录值 |
| 数据库与停写 | 实际数据库路径；停写/停止 API 命令；备份方式、命令、时间、主库及 WAL/SHM 处理方式；备份文件位置和校验值 |
| 迁移与初始化 | 使用该 revision 的 admin 产物执行 migration 的命令；seed/reset 是否执行及原因；禁止使用会重建目标数据库的 reset 代替生产迁移 |
| 正向 smoke | readiness 命令与结果；旧功能查询/认证 smoke；本里程碑应关闭路由的 404 smoke；migration 重复执行和数据保留证据 |
| 回退演练 | 触发条件；切换到兼容 schema-only 产物或停机恢复整库的逐步命令；恢复后的 readiness/查询 smoke；恢复点时间 |
| 数据损失边界 | 切换兼容产物时哪些新能力不可用但数据保留；恢复整库时明确丢弃备份时间之后的全部写入，并记录通知与确认人 |

“切换旧产物”只有在该产物已针对迁移后的 schema 通过 readiness 和旧功能 smoke 时才成立。否则唯一允许的回退流程是迁移前停写、完成整库一致性备份、迁移失败或发布失败后停机恢复整库；不得让仅支持较低 schema 的 binary 尝试打开高版本数据库。development 可使用 `go run ./cmd/admin -- migrate`、`go run ./cmd/admin -- seed` 和 `go run ./cmd/api` 形成 smoke，但发布 Evidence 必须使用待部署的固定产物，而不是工作区源码。

#### P0-3a 本地/CI 发布与恢复模板

当前目标环境明确限定为单进程本地 demo 与 CI 临时目录演练，不存在共享测试环境或生产式部署。v3、v4 使用同一流程模板，但分别保存和验证产物；任何真实共享环境或生产部署都不在本阶段已兑现能力内，启用前必须另行设计部署、备份、权限、监控和恢复流程并重新评审本门禁。

固定产物使用待验证 revision 构建，不提交到仓库；本地默认保存在 `artifacts/phase3/<schema>/<revision>/`，CI 以同一相对结构上传受保留期控制的 artifact。仓库根 `.gitignore` 必须忽略 `/artifacts/`，避免提交 binary、数据库备份、WAL/SHM、hash 清单或原始 smoke 输出；若需要版本化摘要，只提交经过脱敏的独立 Markdown，不提交数据库、sidecar 或原始运行输出。原始证据统一放在 `artifacts/phase3/<schema>/<revision>/evidence/`，checklist 只记录 revision、命令、时间、结果和 artifact 标识。M1-A 的路由关闭由该 revision 根本不注册任何订单 route 保证；M2-B0 保留 M1 GET route，但根本不注册 POST/PATCH route。两者都必须由真实固定产物的 404 smoke 证明，不能仅依赖环境变量或人工约定。PowerShell 构建与摘要模板为：

```powershell
$Revision = (git rev-parse HEAD).Trim()
$Schema = 'v3' # v4 演练时改为 v4
$ArtifactDir = Join-Path 'artifacts/phase3' (Join-Path $Schema $Revision)
New-Item -ItemType Directory -Force $ArtifactDir | Out-Null
go build -trimpath -o (Join-Path $ArtifactDir 'api.exe') ./cmd/api
go build -trimpath -o (Join-Path $ArtifactDir 'admin.exe') ./cmd/admin
Get-FileHash -Algorithm SHA256 (Join-Path $ArtifactDir '*.exe') |
  Select-Object Path, Hash | Format-Table -AutoSize
```

运行配置来自演练任务的环境变量：`APP_ENV=development`、专用 `PORT` 和绝对 `DATA_DIR` 可记录实际值；`JWT_SIGNING_KEY`、demo/bootstrap 密码只记录 secret 来源名称，不记录值。数据库固定为 `<DATA_DIR>/allinme.db`。迁移使用固定 `admin.exe migrate`，启动使用固定 `api.exe`；演练前确认该 `DATA_DIR` 不指向日常开发数据，migration 演练不得使用 reset 代替。实际命令、revision、hash、配置来源、HTTP smoke 输出和执行时间分别记入 M1-1/M2-3 Evidence。

备份必须先优雅停止 API 并确认固定产物进程退出，使 SQLite 不再持有写事务；随后把主库和当时存在的 WAL/SHM 作为一个集合复制到同一空备份目录，并保存 SHA-256 manifest。恢复必须保持 API 停止，先移走或删除目标主库/WAL/SHM，再把备份集合完整复制回 `DATA_DIR`，不得把不同时间的 sidecar 与主库混用。PowerShell 模板为：

```powershell
$DataDir = (Resolve-Path $env:DATA_DIR).Path
$Database = Join-Path $DataDir 'allinme.db'
$BackupDir = Join-Path $ArtifactDir ('backup-' + (Get-Date -Format 'yyyyMMdd-HHmmss'))
$DatabaseFiles = @($Database, "$Database-wal", "$Database-shm") | Where-Object { Test-Path $_ }
New-Item -ItemType Directory -Force $BackupDir | Out-Null
Copy-Item -LiteralPath $DatabaseFiles -Destination $BackupDir
Get-FileHash -Algorithm SHA256 (Join-Path $BackupDir '*') |
  Select-Object Path, Hash | Format-Table -AutoSize

# 恢复时 API 必须仍处于停止状态；$BackupDir 指向已校验的同一备份集合。
Remove-Item -Force -ErrorAction SilentlyContinue $Database, "$Database-wal", "$Database-shm"
Get-ChildItem -File $BackupDir | Copy-Item -Destination $DataDir
```

回退触发条件固定为 migration 失败、readiness 失败、旧功能 smoke 失败、应关闭路由意外可达或迁移后出现数据完整性错误。若对应 schema-only 固定产物已在迁移后 schema 通过 smoke，则切换到该产物并保留迁移后数据；否则停止 API 并恢复迁移前整库集合。整库恢复会丢弃备份完成时刻后的全部写入；本地/CI 演练原则上在停写窗口内完成，因此预期损失为零，但 Evidence 仍必须记录备份时间、恢复点、实际丢弃范围和执行人。恢复后重新运行 readiness、阶段一/二旧功能 smoke、应关闭 route 的 404 smoke，并核对备份 hash。

### M1：只读订单纵切

交付仅含 orders/order_items 的 migration v3、领域模型、只读 repository/service、最小确定性 seed、订单 route metadata、app 最小装配、列表和详情 HTTP。`routeMetadata` 或等价单一事实源在注册第一个订单路由时建立，驱动 mux、method/Allow 和 HEAD 行为；后续 CORS 只消费它，不重构路由。

`NewAuthenticatedAPIWithDependencies` 在 M1 打开一个共享 DB 并同时装配 auth/order service，只注册列表和详情；`NewAPI` 继续不暴露业务路由。构造失败清理 DB 和进程锁。四角色必须能使用真实登录 token 查询 seed 订单。

内部 gate：

- M1-A：完成 additive migration v3、读模型/status 枚举、读取权限、capability、ID/clock、repository contract 和 deterministic seed；开始条件是 P0 范围/契约与 v3 部署/恢复步骤已确认，结束条件是 schema-only binary 已验证 migration 失败事务回滚、重复执行、v3 readiness、无订单路由和 seed/reset；功能回退点为该 v3 schema-only binary；硬门禁为 `go test ./internal/order ./internal/store ./internal/admin ./internal/app -count=1`，race 证据命令为 `go test -race ./internal/order ./internal/store ./internal/admin ./internal/app -count=1`。M1-A 通过前不得注册任何订单 GET/POST/PATCH/Action 路由，不得创建 v4 `idempotency_keys`，也不得实现或放宽金额计算、履约状态转换或写路径。首个可回顾变更推荐按以下顺序拆分，且每一步都不注册订单 HTTP 路由：`internal/order` 领域类型与纯测试 → migration v3 与 repository contract/纯测试 → deterministic development seed/reset → schema-only 构建与 migration/readiness/重复执行/失败回滚/无订单路由 404 smoke。查询 observer/executor、scanner stub、clock、ID generator 和临时数据库 fixture 等测试 seam 应在 M1-A/M1-B 早期先建立，且只放在测试或窄接口中，不扩展生产 HTTP 选项。
- M1-B：完成只读 service/list/detail、M1 route metadata、app wiring 和真实 token 查询；开始条件是 M1-A schema-only binary 已部署并通过 smoke，结束条件是四角色列表/详情 HTTP 通过且将验证矩阵“订单查询”改为 `enabled: yes`；功能回退点为 M1-A binary；硬门禁为 `go test ./internal/httpapi ./internal/app -count=1`，race 证据命令为 `go test -race ./internal/httpapi ./internal/app -count=1`。

开发加速命令（不得作为唯一 gate；使用时必须从 `go test -json` 证明确有测试命中）：

```sh
go test ./internal/order ./internal/store ./internal/admin -run 'Order|List|Get|Seed|Migration' -count=1
go test ./internal/httpapi ./internal/app -run 'Order|Auth|Route|Lifecycle' -count=1
```

### M2：草稿写入纵切

在 M1 的真实 app 上增量注册创建和编辑路由，交付 migration v4、金额校验、写事务、version、创建幂等、snapshot v1、replay/conflict、单连接池等待与 typed Unavailable 适配。operator/admin 可真实创建和编辑；viewer/approver 被拒绝；相同 key 重放和不同 body 冲突通过真实 HTTP 验证。M2-A 不注册 `POST /api/v1/orders` 或 `PATCH /api/v1/orders/{orderId}`；两条写路由只能在完整幂等语义同时就绪的 M2-B1 一次性注册，禁止暴露缺少 snapshot/replay/conflict 的临时 create 契约或提前暴露 edit。

内部 gate：

- M2-A：完成仅在 service/store 包内可测试的 create/edit command、金额/溢出校验、聚合事务和 version 能力；开始条件是 M1-B 已合并，结束条件是 create/edit、权限、输入、金额和 version 包级测试通过，并由真实 `internal/app` 负向测试证明 `POST /api/v1/orders` 与 `PATCH /api/v1/orders/{orderId}` 均为 404、M1 查询仍可用。不得以真实写入 smoke 作为本 gate 结果；功能回退点为 M1-B 只读 API；硬门禁为 `go test ./internal/order ./internal/store ./internal/app -count=1`，race 证据命令为 `go test -race ./internal/order ./internal/store ./internal/app -count=1`。
- M2-B0：只交付 additive migration v4 和 schema compatibility binary，不注册任何订单写路由；开始条件是 M2-A 已合并且 v4 部署/恢复步骤已复核，结束条件是 v3→v4 数据保留、migration 失败回滚、重复执行、v4 readiness、M1 查询 smoke 和 POST/PATCH 仍为 404 均通过。该 binary 是迁移到 v4 后关闭写能力的回退目标；硬门禁为 `go test ./internal/store ./internal/httpapi ./internal/app -count=1`，race 证据命令为 `go test -race ./internal/store ./internal/httpapi ./internal/app -count=1`。
- M2-B1：完成幂等 snapshot、replay/conflict、并发竞争、连接错误分类和 create/edit HTTP，并一次性注册完整 create/edit 外部能力；开始条件是 M2-B0 binary 已部署且 v4 smoke 已通过，结束条件是 operator/admin 真实创建编辑、viewer/approver 拒绝、同 key 只创建一次、replay 不读当前订单、Unavailable 分类有自动化证据，且将验证矩阵“订单写入/幂等”改为 `enabled: yes`；功能回退点为 M2-B0 binary；硬门禁为 `go test ./internal/order ./internal/store ./internal/httpapi ./internal/app -count=1`，race 证据命令为 `go test -race ./internal/order ./internal/store ./internal/httpapi ./internal/app -count=1`。

开发加速命令（不得作为唯一 gate；使用时必须从 `go test -json` 证明确有测试命中）：

```sh
go test ./internal/order ./internal/store -run 'Create|UpdateDraft|Idempot|Conflict|Connection|Busy' -count=1
go test ./internal/httpapi ./internal/app -run 'CreateOrder|UpdateOrder|Idempot|Unavailable' -count=1
```

内部 gate：

具体姓名、替补、日期和升级路径属于执行管理元数据，由团队在项目管理系统维护，不作为产品正确性 checklist 或发布证据；代码评审仍必须有对应 owner/reviewer。

## 9. 非目标与风险

- 不实现退款创建/审批/拒绝；阶段三只保证已支付 seed 和可退款前置数据。
- 不实现附件上传、绑定、下载、看板和页面配置。
- 不加入库存、物流、支付网关、多租户或账号管理。
- 当前代码基线的每个 `store.DB` 使用 `SetMaxOpenConns(1)`：同一 API 进程的第二个事务先等待 database/sql 连接池，不会进入 SQLite busy handler。该值不是阶段三外部硬契约；若实现调整连接池，必须重做连接等待/BUSY 分类证据并保持客户端错误语义。当前路径只验证等待期间尊重 context deadline/cancel，取消后不写入第二个订单。
- SQLite busy/locked 必须用两个独立 `store.DB` 打开同一临时数据库并持有首个写事务来稳定触发；生产 `busy_timeout` 保持 5000ms，测试通过可注入连接选项或等价 seam 使用不超过 500ms 的短 timeout，避免每个错误路径真实等待 5 秒。超时后 repository 返回类型化 unavailable 错误；HTTP handler 使用注入的 order service/repository stub 验证该错误映射为 `503 SERVICE_UNAVAILABLE` 与 `Retry-After: 1`，不要求真实 HTTP 在单连接池内触发 busy。
- 两个独立连接竞争相同 key 时最多创建一个订单；竞争请求允许在首事务及时提交后直接读取并重放 snapshot v1，也允许因 busy/locked 返回 `503`。若返回 `503`，使用同一 key 重试后必须重放首次创建快照；任何路径都不得产生第二个订单。本阶段不扩大连接池，也不增加跨进程“进行中”状态表。
- 单 DB 连接池等待测试与双 DB SQLite BUSY/LOCKED 测试严格分离：前者只断言 channel/barrier 协调下的 context cancel/deadline 原样保留，不映射 503；后者只通过 modernc SQLite 结构化错误类型/错误码识别 BUSY/LOCKED，转换为 typed `Unavailable`，禁止按错误字符串或精确 sleep 判定。
- handler 尚未写响应时收到 `context.Canceled`/`context.DeadlineExceeded`：通过 `http.ErrAbortHandler` 或等价连接中止机制中止连接，不写业务 error envelope、不映射 503；recovery middleware 不得把该取消路径映射为 `500 INTERNAL_ERROR`。access log 增加稳定结构化字段 `outcome`，正常已提交响应为 `completed`，未提交响应的取消为 `canceled`，typed `Unavailable` 响应为 `unavailable`；`status` 在已提交响应时记录实际 HTTP status，未提交的 canceled 请求省略 `status`，不得沿用 tracking writer 的默认 200。测试按 slog attribute 断言字段，不依赖日志文案。保留 `errors.Is`；已提交响应后的取消不尝试改写 status 或 body，并按实际已提交 status 记录。测试必须使用真实 `httptest.Server` 或连接级客户端观察连接中止，不能只断言 `httptest.ResponseRecorder`。
- snapshot v1 是创建时不可变持久化 DTO，不是当前订单缓存；后续编辑/Action 不更新它。v5 起完整 snapshot JSON 的 SHA-256 保存在独立 `snapshot_digest` 列，重放必须先核对摘要，因此迁移完成后再替换为另一个格式合法且不重复的 item ID 也会安全失败。v5 在同一迁移事务中对 v4 历史 snapshot JSON 计算并回填摘要，使结构与业务事实合法的历史记录继续按原契约重放；已损坏的历史 snapshot 仍在严格解码和事实校验阶段返回 typed internal，不回退为当前订单查询或重新创建。
- 阶段三不提供在线幂等记录清理，development reset 继续通过重建数据库清除。snapshot v1 的保留、清理、归档和版本迁移责任归属阶段四后端负责人，并作为阶段四开工前置契约；K4 必须记录该责任和后续阶段。进入阶段四或生产化前必须另行冻结按 endpoint/operation 的保留期、归档或清理策略，以及 snapshot 版本兼容迁移规则；在此之前不得后台静默删除或原地重写 snapshot v1。
- 所有连接池等待、双 DB BUSY/LOCKED、并发幂等、并发 Action 和真实连接中止测试都使用 `context.WithTimeout` 或测试级 deadline，单测试上限为 10 秒；cleanup 必须释放 barrier/channel、回滚未提交事务、关闭所有 DB/server 并删除临时目录。超时必须使测试失败并完成清理，不得让 CI 无限等待。
- 搜索不承诺完整 Unicode case folding；测试必须包含 ASCII 大小写和中文原值 fixture，避免把 SQLite 默认 `NOCASE` 误描述为 Unicode 能力。

repository/service 错误分类固定如下：幂等唯一键冲突必须重新读取记录并按 digest 分类为 replay 或 `IDEMPOTENCY_CONFLICT`；只有驱动分类为 SQLite BUSY/LOCKED 才转换为内部 typed `Unavailable`；`context.Canceled`/`context.DeadlineExceeded` 原样保留，不转换为 503；关闭数据库、其他 SQL/扫描错误、损坏数据和 snapshot 错配转换为 typed internal error。handler 只把 `Unavailable` 映射为 `503` + `Retry-After: 1`，普通事务失败不得自动重试创建。

JSON 数字边界固定为可观察词法行为：quantity、unitPrice、version 只接受十进制整数词法 `^-?(0|[1-9][0-9]*)$`；`1.0`、`1e2`、字符串、null 和超出 int64 均为 `400 INVALID_REQUEST`，解析成功后的范围错误才进入 `422 VALIDATION_FAILED`。实现可使用 `json.Number` 或等价无精度损失 decoder，不得先转 float64 再恢复整数。

查询时间在 HTTP 边界使用 `time.Parse(time.RFC3339)`，例如 `2026-01-01T00:00:00+08:00` 规范化为 `2025-12-31T16:00:00Z`；repository 接收 `time.Time` 并按时间瞬间比较，响应和数据库订单时间始终输出 UTC `Z`，不得直接按原始 offset 字符串比较。

自动化证据固定使用 store 内部 executor/query observer 断言列表空页执行 COUNT + page 两次查询，非空页再执行一次批量明细完整性查询，且无逐订单查询；用协调点和第二个 DB 证明三者共享同一 snapshot。扫描失败通过 store 包内测试专用 row scanner/executor stub 注入，不向生产 repository 接口增加故障开关；损坏数据通过临时 SQLite fixture 在测试连接上启用 `PRAGMA ignore_check_constraints=ON` 后直接插入非法枚举、金额、时间、UTF-8、position、明细数量或聚合总额，随后恢复约束并验证 repository 安全分类，禁止修改正式 migration 来制造坏数据。分别覆盖单连接池等待、双 DB BUSY/LOCKED、typed Unavailable 的 HTTP 503 映射。测试名称和内部 seam 名可替换，checklist Evidence 必须记录实际名称与命令，不能只写“代码审查通过”。文档门禁仍使用 `docs/audit/validate.ps1` 检查 frontmatter、相对链接与 `git diff HEAD --check`，`validate.tests.ps1` 验证合法 fixture 成功、缺失链接 fixture 失败和 fragment 只校验文件目标。

## 10. 完成标准

阶段三硬门禁：

- 九个订单 endpoint 具备当前 API 文档、真实 HTTP 集成测试和稳定错误契约；
- 查询、金额、状态机、权限、version 和幂等均有 repository/service 测试；
- 全部合法履约转换和主要非法转换有证据，viewer/approver 的写权限被后端拒绝；
- seed 可重复并覆盖阶段四所需订单/支付状态；
- 认证应用完成订单装配，基础 API 不暴露订单路由；CORS 配置失败无启动副作用；
- 最小跨源 OPTIONS + 带认证订单请求 smoke 通过，但不宣称页面联调完成；
- 订单查询、订单写入/幂等、订单履约 Action 验证矩阵分别在 M1-B、M2-B1、M3-A 的同一变更中改为 `enabled: yes`；
- CORS 验证矩阵在 M3-B 后独立改为 `enabled: yes`，不作为订单或幂等能力启用的前置条件；
- 全量 test、vet、race 和文档检查通过后，阶段三 checklist 工程项全部勾选并提交完成报告。

阶段六门禁独立包含页面 YAML、L0-L4 校验、页面 action/reaction、完整 Schema-UI 浏览器场景和页面级回归证据；这些未完成不阻塞阶段三归档，也不阻塞订单 endpoint 迁入当前 API，更不得由阶段三最小跨源 smoke 冒充完成。

归档是完成报告之后的后置动作，不属于 checklist：先汇报已完成项、未执行项和剩余风险，只有用户明确确认后，才将 `0003` 计划/checklist 移入 `docs/audit/archived/` 并更新活跃与归档索引。
