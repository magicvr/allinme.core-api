# 阶段四：退款与经营看板 Checklist

配套计划：[阶段四退款与经营看板开发计划](./0004-2026-07-13-plan.md)。领域事实以[领域模型](../05-domain-model.md)为准，endpoint 范围以[目标 HTTP API](../03-http-api-target.md)为准，验证启用状态以[验证规则](../04-validation.md)为准。本 checklist 只记录实施门禁和实际 Evidence。

统一要求：每项完成后在下一行填写日期、类型、命令/测试、revision 或 smoke 结果。带 `-run` 的命令只用于开发加速，不能替代对应包级硬门禁。P0、M1/M2/M3 的功能正确性、并发、幂等、HTTP 契约和包级 test/race 是开发硬门禁；schema-only/旧 binary 恢复、query plan、完整 smoke、远端 CI 和归档记录是发布证据，可在对应功能切片稳定后集中完成，但发布前不得省略。并发测试必须设置 deadline 并释放 goroutine、事务、DB、server 和临时目录；日志和失败输出不得包含 Authorization、token、密码、完整请求 body 或幂等 snapshot。

- [ ] M1-1. additive migration v6 创建 `refunds`、独立退款幂等表和必要索引并设置 `user_version=6`；覆盖空库、v5 升级、重复 migration、失败回滚、旧订单/幂等数据保留及 readiness。
  Evidence:
- [ ] M1-2. 退款 ID、状态、金额、原因、version、审计字段和 capability 通过纯领域测试；领域层覆盖 `utf8.ValidString`、NUL、经 `strings.TrimSpace` 后 1..500 bytes、允许并保留内部换行；创建时 `createdAt == updatedAt` 且未决定，approve/reject 时 `updatedAt == decidedAt`，失败、冲突和重放不更新时间。原始非法 UTF-8 body 的 transport 门禁由 M2-2 覆盖。
  Evidence:
- [ ] M1-3. development `refund_demo=1` 精确创建并验证 P0-14 的固定订单、明细、退款、actor、金额、时间、版本和 dashboard 快照；执行顺序为 `runtime → auth_demo → order_demo → refund_demo`，重复 seed 只读验证且允许额外业务数据，production 不写 refund_demo，reset 恢复确定性基线。
  Evidence:
- [ ] M1-4. create miss 按订单存在性 → orderVersion → order writer fence 后复核 → paymentStatus → amount/availableRefundAmount 分类；分别稳定返回 404、VERSION_CONFLICT、STATE_CONFLICT、带 amount detail 的 VALIDATION_FAILED。fence 不增加订单 version/updatedAt，失败不生成退款或幂等记录。
  Evidence:
- [ ] M1-5. canonical digest 固定编码 `operation/orderId/amount/reason/orderVersion`；JSON 空白/字段顺序和 reason 外围空白等价，事实差异冲突。snapshot v1 精确保存首次 201 Refund DTO 且 capability 固定 false；命中先于当前 orderVersion/paymentStatus/availableRefundAmount 检查，审批后仍重放首次 snapshot；未知版本或损坏返回 INTERNAL_ERROR 且不重新创建。
  Evidence:
- [ ] M1-6a. 两个独立 DB 句柄使用不同 key 并发申请时，`PENDING + COMPLETED` 不得超过订单总额；锁竞争 loser 如先返回 unavailable，重试后必须得到确定的金额校验失败或成功结果。
  Evidence:
- [ ] M1-6b. 两个独立 DB 句柄使用相同 key 并发申请时最多创建一条退款和一份 snapshot，两方最终得到语义等价的首次创建结果，不得把不同 key 超额测试替代本项。
  Evidence:
- [ ] M1-7. approve 按退款存在性 → version → PENDING → self-approval → order writer fence 后复核 refund/order → 关联订单完整性 → completed 金额/paymentStatus 一致性 → 双条件更新分类；refund 使用 `id+version+PENDING` CAS，order 使用 fence 后观察到的 `id+version` CAS，任一 0 row 整体回滚。order CAS 竞争稳定映射可重试 unavailable，原输入重试后重新观察聚合；成功在单一事务更新 refund、订单 paymentStatus/version/updatedAt。
  Evidence:
- [ ] M1-8. reject 按退款存在性 → version → PENDING → self-approval → 条件更新分类，不读取或校验订单；仅更新退款及审批审计字段，不修改订单 paymentStatus/version，终态不可再次转换，失败不修改时间；后续订单读取因 pending 消失而使 availableRefundAmount 回升。
  Evidence:
- [ ] M1-9. 同一 refund version 的跨连接 approve/reject 竞争最多一个成功；loser 可先得到 typed unavailable，但 winner 提交后使用原 version 重试必须稳定得到 VERSION_CONFLICT，最终不能重复退款或改写终态。
  Evidence:
- [ ] M1-10. 订单领域和 repository 返回 `availableRefundAmount` 并支持动态 `canRequestRefund`：角色允许、paymentStatus 可退款且金额大于零；订单列表在同一只读 snapshot 中批量加载当前页退款占用，禁止逐订单查询，详情复用同一规则，`canApproveRefund` 恒为 `false`。
  Evidence:
- [ ] M1-11. DRAFT edit 先按输入 version 取得 order writer fence，再在同一事务验证旧 paymentStatus/旧 total/退款历史一致并计算 checked `PENDING + COMPLETED`；newTotal 过低返回冻结 items detail 并完整回滚，合法 edit 按新总额更新 paymentStatus 且 order version 只加一，损坏聚合返回 INTERNAL_ERROR 且不得被 edit 修复。
  Evidence:
- [ ] M1-12. 两个独立 DB 句柄竞争 edit/refund create：refund 先提交时 edit 重新观察占用并成功或稳定校验失败；edit 先提交时旧 orderVersion 的 refund 稳定冲突；最终金额、paymentStatus、version、退款数和明细保持一致。

## M2. 退款 HTTP 闭环

- [ ] M2-1. `GET /api/v1/refunds` 只允许 approver/admin，仅接受 status/orderId/page/pageSize，拒绝 sort/order/未知/重复参数；默认 page=1/pageSize=20、最大 100、固定 createdAt/id 降序。COUNT、page rows 和 actor hydration 必须共享显式只读事务，actor 批量加载且无逐退款查询；返回冻结 page envelope、actor/null DTO 和 capability；operator 返回 403。
  Evidence:
- [ ] M2-2. create 严格按 route/method → CORS → auth → role → Content-Type → 原始 body UTF-8 → key → orderId → JSON/整数词法 → typed normalization → idempotency lookup → create miss 业务分类执行并验证组合错误优先级；真实非法 UTF-8 bytes 固定为 `400 INVALID_REQUEST`，不得写退款或幂等记录。
  Evidence:
- [ ] M2-3. approve/reject 严格按 route/method → CORS → auth → role → Content-Type → refundId → JSON/整数词法 → typed version → 各自业务分类执行；成功固定返回 200 当前 Refund DTO，验证 version 优先于 state、state 优先于 self-approval，reject 不受订单异常 paymentStatus 影响。
  Evidence:
- [ ] M2-4. 真实 JWT 集成测试证明 operator/admin 可申请，approver/admin 可审批他人申请，viewer/approver 不能申请，operator/viewer 不能审批，admin 不能自批。
  Evidence:
- [ ] M2-5. route metadata、mux 和 CORS 使用同一组已注册路由；`HEAD` 返回 405 且 `Allow` 不含 HEAD，合法 CORS preflight `OPTIONS` 返回 204，带 Origin 但缺失/非法 requested method 返回 403，未启用 CORS或不带 Origin 的普通 `OPTIONS` 返回 405，unknown path 保持 404。
  Evidence:
- [ ] M2-6. 真实 app 完成申请、幂等重放、不同 body 冲突、审批、审批后再次重放原创建 snapshot、拒绝和列表刷新；重启后结果保持，reset/seed 恢复基线。
  Evidence:
- [ ] M2-7. 所有 Order DTO 的 `availableRefundAmount` 和 `canRequestRefund` 保持一致；PENDING 占用后金额及 capability 立即变化，列表使用固定批量查询而非 N+1，详情/create/edit/Action 字段语义一致，`canApproveRefund` 保持 false。
  Evidence:
- [ ] M2-8. 设置 `DisableRefundRoutes` 后阶段三订单查询/写入/履约仍可用，退款 list/create/approve/reject 全部 404；`DisableDashboardRoutes` 不影响退款，两个字段不从运行配置加载。本项仅是代码级兼容性证据，不声称已部署 binary 可运行时切换。
