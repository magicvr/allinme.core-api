# 阶段四：退款与经营看板 Checklist

配套计划：[阶段四退款与经营看板开发计划](./0004-2026-07-13-plan.md)。领域事实以[领域模型](../05-domain-model.md)为准，endpoint 范围以[目标 HTTP API](../03-http-api-target.md)为准，验证启用状态以[验证规则](../04-validation.md)为准。本 checklist 只记录实施门禁和实际 Evidence。

统一要求：每项完成后在下一行填写日期、类型、命令/测试、revision 或 smoke 结果。带 `-run` 的命令只用于开发加速，不能替代对应包级硬门禁。并发测试必须设置 deadline 并释放 goroutine、事务、DB、server 和临时目录；日志和失败输出不得包含 Authorization、token、密码、完整请求 body 或幂等 snapshot。

## P0. 契约与基线

- [ ] P0-1. 交叉复核领域模型、目标 API、路线图、验证矩阵和 plan，冻结阶段四范围及事实源边界；附件、页面、真实支付和生产分析明确为非目标。
  Evidence:
- [ ] P0-2. 冻结 `PENDING` 是否占用额度、可退金额公式、paymentStatus 映射、订单/refund version 更新时间和 self-approval 规则，并同步领域模型。
  Evidence:
- [ ] P0-3. 在阶段四 plan 冻结退款 create/list/approve/reject 的 body、query、DTO、状态码、错误码、短路顺序和大小限制；目标 API 只同步 endpoint 范围、角色和跨阶段边界，详细契约待实现及集成门禁通过后迁入当前 API。
  Evidence:
- [ ] P0-4. 冻结退款幂等作用域、normalized digest、独立 snapshot v1、损坏记录行为和保留/清理策略；确认不改变阶段三订单 snapshot 语义。
  Evidence:
- [ ] P0-5. 冻结看板 `orderCount/grossAmount/completedRefundAmount/netAmount`、paymentStatus 纳入集合、履约状态影响、UTC 日期窗口、日期归属和补零规则；同步领域模型并消除“成交额”重复扣减歧义。
  Evidence:
- [ ] P0-6. 冻结订单 `canRequestRefund` 的角色/paymentStatus/availableAmount 规则，订单 `canApproveRefund` 继续为 `false`；冻结当前页批量退款占用查询及列表/详情一致性要求。
  Evidence:
- [ ] P0-7. 冻结测试装配字段 `DisableRefundRoutes`、`DisableDashboardRoutes`；确认两者不是环境配置或生产 feature flag，并精确定义各自关闭的 route 集合。
  Evidence:
- [ ] P0-8. 记录开工 revision、Go 版本、`go test ./... -count=1`、`go vet ./...`、`go test -race ./... -count=1` 和文档 validator 基线。

## M1. 退款核心与 Schema

- [ ] M1-1. additive migration 创建 `refunds`、独立退款幂等表和必要索引；覆盖空库、v5 升级、重复 migration、失败回滚、旧订单/幂等数据保留及 readiness。
  Evidence:
- [ ] M1-2. 退款 ID、状态、金额、原因、version、审计字段和 capability 通过纯领域测试；创建时 `createdAt == updatedAt` 且未决定，approve/reject 时 `updatedAt == decidedAt`，失败、冲突和重放不更新时间。
  Evidence:
- [ ] M1-3. development seed 覆盖可退、部分退款、全额退款、待审批和已拒绝场景；重复 seed 只读验证，production 不写退款，reset 恢复确定性基线。
  Evidence:
- [ ] M1-4. 创建退款校验订单存在性、orderVersion、支付状态、金额和占用额度；失败不生成退款或幂等记录。
  Evidence:
- [ ] M1-5. 退款幂等 normalized digest 包含规范化 orderId，snapshot v1 对相同请求稳定重放，对不同订单或不同 body 返回冲突；命中后先于当前 orderVersion、paymentStatus 和 availableAmount 检查返回首次 snapshot，审批后再次重放仍成功；未知版本、摘要不匹配、JSON 损坏和元数据错配安全失败且不重新创建。
  Evidence:
- [ ] M1-6. 两个独立 DB 句柄并发申请不会使 `PENDING + COMPLETED` 超过订单总额，同一 key 最多创建一条退款和一份 snapshot。
  Evidence:
- [ ] M1-7. approve 在单一事务中完成申请审批隔离、refund version/state 更新、completed 金额复核、订单 paymentStatus/version/updatedAt 更新；任一步失败整体回滚。
  Evidence:
- [ ] M1-8. reject 仅更新待处理退款及审批审计字段，不修改订单 paymentStatus/version；approve/reject 终态不可再次转换，成功使用一次注入 clock，失败不修改 `updatedAt/decidedAt`。
  Evidence:
- [ ] M1-9. 同一 refund version 的跨连接 approve/reject 竞争最多一个成功；另一请求得到稳定 version/state conflict 或 typed unavailable，不能重复退款。
  Evidence:
- [ ] M1-10. 订单领域和 repository 支持动态 `canRequestRefund`：角色允许、paymentStatus 可退款且 availableAmount 大于零；订单列表在同一只读 snapshot 中批量加载当前页退款占用，禁止逐订单查询，详情复用同一规则，`canApproveRefund` 恒为 `false`。
  Evidence:
- [ ] M1-11. schema-only app 在新 schema 上保持阶段一至三 API 可用，四条退款路由和三个看板路由仍为 404；记录路由关闭和数据库恢复回退证据。
  Evidence:
- [ ] M1-12. `go test ./internal/order ./internal/store ./internal/admin ./internal/app -count=1` 及对应非缓存 race 门禁通过。
  Evidence:

## M2. 退款 HTTP 闭环

- [ ] M2-1. `GET /api/v1/refunds` 实现冻结的筛选、分页、稳定排序、DTO 和 capability，并在一致只读 snapshot 中返回结果与 total。
  Evidence:
- [ ] M2-2. `POST /api/v1/orders/{orderId}/refunds` 严格处理 method、CORS、authentication、role、Content-Type、Idempotency-Key、orderId、JSON、typed validation 和 repository 错误优先级。
  Evidence:
- [ ] M2-3. approve/reject Action 严格处理 refundId、version 整数词法、状态/版本冲突、self-approval、取消和 unavailable，成功返回冻结 DTO。
  Evidence:
- [ ] M2-4. 真实 JWT 集成测试证明 operator/admin 可申请，approver/admin 可审批他人申请，viewer/approver 不能申请，operator/viewer 不能审批，admin 不能自批。
  Evidence:
- [ ] M2-5. route metadata、mux 和 CORS 使用同一组已注册路由；`HEAD` 返回 405 且 `Allow` 不含 HEAD，合法 CORS preflight `OPTIONS` 返回 204，带 Origin 但缺失/非法 requested method 返回 403，未启用 CORS或不带 Origin 的普通 `OPTIONS` 返回 405，unknown path 保持 404。
  Evidence:
- [ ] M2-6. 真实 app 完成申请、幂等重放、不同 body 冲突、审批、审批后再次重放原创建 snapshot、拒绝和列表刷新；重启后结果保持，reset/seed 恢复基线。
  Evidence:
- [ ] M2-7. 真实订单列表、详情和退款创建后的 `canRequestRefund` 与 availableAmount 保持一致；PENDING 占用后及时变为 false 或反映剩余额度，列表使用固定批量查询而非 N+1，`canApproveRefund` 保持 false。
  Evidence:
- [ ] M2-8. 设置 `DisableRefundRoutes` 后阶段三订单查询/写入/履约仍可用，退款 list/create/approve/reject 全部 404；`DisableDashboardRoutes` 不影响退款，两个字段不从运行配置加载。
  Evidence:
- [ ] M2-9. `go test ./internal/order ./internal/store ./internal/httpapi ./internal/app ./internal/admin -count=1` 及对应非缓存 race 门禁通过；验证矩阵“退款”改为 `enabled: yes`。
  Evidence:

## M3. 经营看板

- [ ] M3-1. summary 返回 `orderCount/grossAmount/completedRefundAmount/netAmount/currency`；orderCount 统计全部订单，gross 只纳入 `PAID/PARTIALLY_REFUNDED/REFUNDED` 原始金额且不受履约状态影响，net 精确为 gross 减 completed refund，checked arithmetic 防止溢出。
  Evidence:
- [ ] M3-2. order-status 返回全部订单状态及数量，固定顺序并对缺失状态补零；结果与订单查询交叉一致。
  Evidence:
- [ ] M3-3. trend 只接受 `days=7|30`，使用注入 clock 的 UTC 今天形成连续窗口，逐日补零并返回 `orderCount/grossAmount/completedRefundAmount/netAmount`；orderCount 包含全部订单并按 createdAt 归属，gross 只包含冻结 paymentStatus 集合并按 createdAt 归属，完成退款按 decidedAt 归属。
  Evidence:
- [ ] M3-4. dashboard repository 在显式只读事务中完成相关聚合；同一响应不得混合不同 SQLite commit，取消、关闭 DB、损坏数据和扫描错误返回稳定错误。
  Evidence:
- [ ] M3-5. 四角色真实 token 均可读取三个看板 endpoint；未认证、错误 method/query、CORS、取消和 unavailable 行为稳定。
  Evidence:
- [ ] M3-6. 固定 seed 快照证明三个看板响应与订单/退款查询一致；完成一笔退款后，`completedRefundAmount`、`netAmount`、订单 paymentStatus 和 version 同步变化，退款不会被重复扣减。
  Evidence:
- [ ] M3-7. injectable clock 边界测试覆盖 UTC 午夜、窗口首日、窗口外订单和窗口外退款，7/30 日窗口不依赖真实当前时间。
  Evidence:
- [ ] M3-8. 代表性 fixture 的索引与 query plan 已复核，不存在逐订单/逐退款查询；本项只作退化保护，不声明生产 SLA。
  Evidence:
- [ ] M3-9. 设置 `DisableDashboardRoutes` 后退款与阶段三 API 保持可用，三个看板 endpoint 全部 404；`DisableRefundRoutes` 不影响看板，两个字段不从运行配置加载。
  Evidence:
- [ ] M3-10. `go test ./internal/order ./internal/store ./internal/httpapi ./internal/app -count=1` 及对应非缓存 race 门禁通过；验证矩阵“看板”改为 `enabled: yes`。
