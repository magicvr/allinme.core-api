# 阶段三：订单查询与履约开发计划

配套路线：[实施路线图](../06-implementation-roadmap.md)。本阶段只实现订单查询、订单写入和履约状态变化；退款、附件、看板、页面配置和账号管理不进入本轮实现。

## 1. 目标与完成边界

在阶段二认证主体之上，建立可持久化的订单核心，使四类角色能够通过真实 HTTP API 完成：

- `viewer`、`operator`、`approver`、`admin` 查看订单列表与详情；
- `operator`/`admin` 创建草稿、编辑草稿并推进确认、履约、发货、完成或取消；
- 列表支持关键词、订单状态、支付状态、创建日期、allowlist 排序和服务端分页；
- 所有金额由订单项以整数最小货币单位计算，服务端不接受客户端总额作为事实；
- 创建型请求支持幂等键，状态变化和编辑使用 `version` 乐观锁；
- 响应提供基于当前主体和资源状态计算的 `canEdit`、`canAdvance`、`canCancel` 展示字段；阶段三尚无退款 API，因此 `canRequestRefund`、`canApproveRefund` 恒为 `false`，所有 capability 字段均不替代后端授权；
- development seed 覆盖订单状态和支付状态，为阶段四退款提供前置数据。

完成本阶段不代表页面已实现。Schema-UI 页面配置属于阶段六；本阶段只提供页面所需的稳定 API 和集成证据。

## 2. 已冻结的领域规则

订单履约状态严格为：`DRAFT -> CONFIRMED -> FULFILLING -> SHIPPED -> COMPLETED`；`DRAFT`、`CONFIRMED`、`FULFILLING` 可取消，终态不可取消。创建/编辑只允许 `operator` 和 `admin`，查看允许全部角色。创建后的 `version` 初值固定为 `1`，编辑或 Action 成功后原子 `+1`。`createdAt` 创建后不可变；成功 edit/Action 使用注入 clock 设置新的 `updatedAt`，失败请求和幂等重放不修改任一时间字段。

编辑和 Action 采用“事务内读取当前资源 → 校验 version 与状态 → 同时匹配 ID、源状态和 version 的条件更新”。首次读取或条件更新后的重新读取固定按存在性 → version → 状态的顺序分类：资源不存在为 `NOT_FOUND`，资源存在但请求 version 不同为 `VERSION_CONFLICT`，version 相同但源状态不允许为 `STATE_CONFLICT`；version 与状态同时不匹配时返回 `VERSION_CONFLICT`。并发导致条件更新未命中时不得一律映射为版本冲突，也不得向客户端暴露 SQL 行数或底层 SQLite 错误。

支付状态的数据库枚举固定允许 `UNPAID`、`PAID`、`PARTIALLY_REFUNDED`、`REFUNDED`，避免阶段四修改枚举约束；阶段三创建写路径只能产生 `UNPAID`，阶段三 seed 只写 `UNPAID` 或 `PAID`，任何 HTTP 写路径均不得写入后两个退款状态。订单币种首版只允许 `CNY`。金额使用 `int64` 表示分，`unitPrice` 固定为 `1..9_999_999_999`，订单 `totalAmount` 固定为 `0..9_999_999_999`，数量固定为 `1..10_000`；每次乘法和加法先执行 checked arithmetic，再校验总额上限，数据库同时以 CHECK 保护范围。

capability 精确语义固定如下：`canEdit` 仅对 `operator`/`admin` 的 `DRAFT` 为 `true`；`canAdvance` 仅对 `operator`/`admin` 且存在唯一下一步履约 Action 时为 `true`，即 `DRAFT`、`CONFIRMED`、`FULFILLING`、`SHIPPED`，`COMPLETED`/`CANCELLED` 和 viewer/approver 恒为 `false`；`canCancel` 独立计算，仅对 `operator`/`admin` 的 `DRAFT`、`CONFIRMED`、`FULFILLING` 为 `true`；阶段三 `canRequestRefund`、`canApproveRefund` 恒为 `false`。

订单和明细 ID 分别固定为 `ord_`/`itm_` 加 32 个小写十六进制字符，来自 128-bit `crypto/rand`，不得由时间、客户、数据库行号或数量推导。development seed 也使用相同格式的固定 hex ID；固定性只用于可重复 seed，不放宽 HTTP path 校验。时间由注入 clock 生成，repository 写入前调用 `UTC()` 并固定以 Go `time.RFC3339` 的 `Z` 形式保存和返回；SQLite CHECK 只验证文本形状与 `Z` 后缀，真实可解析性和 UTC 规范化由 repository 测试保证。

数据库使用 `orders`、`order_items`、`idempotency_keys` 三类业务表。`order_items` 保存从 `0` 开始、订单内唯一的 `position`，详情固定按 `position ASC` 返回；外键使用 `order_id REFERENCES orders(id) ON DELETE CASCADE`。阶段三不提供删单 API，幂等记录不随订单单独清理，development reset 通过重建数据库统一清除。订单 service 通过窄 repository 接口协调事务、权限、状态、version、金额和幂等；repository 不决定状态转换，handler 不直接执行 SQL 或拼装领域规则。

订单 service 直接接收 `auth.Principal` 并使用 `auth.RoleAllowed(principal.Role, auth.RoleOperator, auth.RoleAdmin)`；不得在 `internal/order` 重新声明 role、复制角色字符串或建立隐式角色排序。`admin` 是订单写能力的显式允许角色，与领域模型和目标 API 保持一致。

## 3. 已冻结的 HTTP 与幂等契约

| 项 | 阶段三冻结值 |
|---|---|
| 创建幂等 header | `Idempotency-Key`；必填，长度 `1..128` bytes，正则 `[A-Za-z0-9][A-Za-z0-9._:-]{0,127}`；缺失或非法返回 `400 INVALID_REQUEST`，`error.details[0].field` 为 `Idempotency-Key` |
| 幂等适用范围 | 仅 `POST /api/v1/orders`；`PATCH` 和五个 Action 即使携带 `Idempotency-Key` 也明确忽略该 header，只使用 `version`，不得返回 400 或写幂等记录 |
| 幂等作用域 | 数据库唯一键固定为 `principal user ID + method + route/operation + Idempotency-Key`。阶段三只有 `POST /api/v1/orders` 一个幂等 endpoint；相同主体、method、route/key 的不同 normalized request digest 返回 `409 IDEMPOTENCY_CONFLICT`。阶段四新增幂等 endpoint 时可复用同一 key 而不与订单创建冲突；method/route 仍作为记录完整性元数据核对，错配视为损坏记录并返回 `500 INTERNAL_ERROR` |
| 幂等存储结果 | 保存 `snapshot_version=1` 与独立 snapshot v1 JSON；它表示首次创建成功时 version=1 的订单结果，不查询或拼接当前订单。首次与重放返回语义等价的同一快照，即使订单后来已编辑或推进也不改变；未知版本、损坏 JSON 或元数据错配返回 `500 INTERNAL_ERROR`，绝不重新创建 |
| snapshot v1 DTO | 顶层仅含 `order`；order 固定包含字符串 `id/customerName/status/paymentStatus/currency/createdAt/updatedAt`、int64 `totalAmount/version`、数组 `items`；item 固定包含字符串 `id/sku/name` 与 int64 `quantity/unitPrice`。items 按 position 排序但不保存 position；不保存 capability、request ID、HTTP status/header 或 error envelope |
| snapshot 重放映射 | service 从 snapshot v1 恢复创建结果，再按当前已认证 principal 与快照中的 `DRAFT` 状态重新计算 capability；由于 create 只允许 operator/admin，两者 capability 相同。不得查询当前 orders/order_items 补字段，响应中的订单事实字段必须来自不可变快照 |
| snapshot 元数据校验 | 幂等记录独立列固定保存并在读取时核对 principal user ID、key、method、route/operation、request digest、order ID、snapshot version；阶段三记录必须为 method=`POST`、route=`/api/v1/orders`。snapshot 内 order ID 必须与列值相同，status/paymentStatus/version 必须为 `DRAFT`/`UNPAID`/`1`，`createdAt == updatedAt`，时间/金额/枚举/items 必须通过 snapshot v1 解码校验 |
| 请求摘要 | 严格解码后先将 customerName/SKU/item name trim、校验 currency/数值并形成 normalized typed command，再按固定字段顺序编码 canonical JSON；保留明细顺序，忽略原始 JSON 空白、对象字段顺序和字符串边缘空白，因此 `"Alice"` 与 `" Alice "` 使用同一摘要；摘要为 SHA-256，并包含固定操作标识 `POST /api/v1/orders` |
| Content-Type | create/edit/Action 必须提供可由 `mime.ParseMediaType` 解析且主类型为 `application/json` 的 Content-Type，可携带合法参数；缺失、其他媒体类型或非法值返回 `415 UNSUPPORTED_MEDIA_TYPE` |
| JSON body 上限 | create/edit 为 `64 KiB`，Action 为 `1 KiB`；超限返回 `400 INVALID_REQUEST`，未知字段和多余 JSON 值同样拒绝 |
| JSON 命名与 create body | 所有 JSON 字段使用 camelCase；create 固定为 `{"customerName":"...","currency":"CNY","items":[{"sku":"...","name":"...","quantity":1,"unitPrice":100}]}`，不接受客户端 ID、total、status、paymentStatus 或 version |
| edit / Action body | edit 为与 create 相同三个事实字段外加整数 `version`；五个 Action body 仅允许 `{"version":n}`，不接受目标状态或其他字段 |
| 字符串与明细 | `customerName` trim 后为 `1..120` UTF-8 bytes；`sku` trim 后为 `1..64` bytes；item `name` trim 后为 `1..160` bytes；保存 trim 后值；`items` 数量为 `1..100`，按请求数组顺序分配 `position` |
| currency | create/edit 必填且只能精确为 `CNY`，不提供缺省值；缺失或其他值为 `422 VALIDATION_FAILED` |
| version | 创建为 `1`；每次成功 edit/Action 原子 `+1`；edit/Action 缺失、null、字符串、小数或超出 int64 属于 JSON 结构错误并返回 `400 INVALID_REQUEST`，解析成功但 `version <= 0` 返回 `422 VALIDATION_FAILED`，正整数但与资源不一致返回 `409 VERSION_CONFLICT` |
| 分页 | query 为 `page`、`pageSize`；默认 `1`/`20`，`page >= 1`，`pageSize=1..100` |
| 筛选 query | `q`、`status`、`paymentStatus`、`createdFrom`、`createdTo`；`q` trim 后最多 200 UTF-8 bytes；日期接受 Go `time.RFC3339` 的 `Z` 或合法 offset，解析后规范化为 UTC，范围为闭区间；重复参数拒绝 |
| 排序 | query 为 `sort`、`order`；allowlist 为 `createdAt`、`updatedAt`、`totalAmount`、`customerName`、`status`，方向仅 `asc`/`desc`，默认 `createdAt desc`；所有排序固定追加 `id` 同方向作为唯一二级排序，即 `ORDER BY <allowlisted column> <direction>, id <direction>` |
| 列表 envelope | `{"items":[],"total":0,"page":1,"pageSize":20}`；`total` 是应用全部筛选后的总数；COUNT 与当前页查询必须在同一只读事务中执行，保证来自同一 SQLite snapshot |
| Order 响应 DTO | 固定包含 `id/customerName/status/paymentStatus/currency/totalAmount/version/createdAt/updatedAt/canEdit/canAdvance/canCancel/canRequestRefund/canApproveRefund`；所有字段 camelCase，时间为 UTC RFC 3339 |
| Item 响应 DTO | 详情/create/edit/Action 中每项固定包含 `id/sku/name/quantity/unitPrice`；`position` 仅用于稳定排序，不进入 JSON；列表项不含 `items` 字段 |
| 成功状态 | create 首次为 `201` + 首次创建的完整订单结果，重放为 `201` + 语义等价的 snapshot v1；edit 与五个 Action 均为 `200` + 本次成功写入后的当前订单。列表项不含 `items`，详情/create/edit/Action 响应包含按 `position` 排序的 `items` |
| orderId path | 仅接受 `ord_` + 32 位小写 hex；格式非法与格式合法但不存在均返回 `404 NOT_FOUND`，避免暴露 ID 校验细节 |
| 详情附件 | 阶段三不返回 `attachments` 字段，也不返回附件 ID、URL 或附件 capability；阶段五实现时新增并冻结该字段 |
| 错误 details | `error.details` 为可选数组，元素固定为 `{"field":"items[0].quantity","message":"must be between 1 and 10000"}`；无字段错误时省略整个 `details`，不返回 `[]`/`null`；JSON 字段用 camelCase path，query 用参数名如 `pageSize`，header 用规范 header 名 `Idempotency-Key`；元素按校验顺序稳定返回，message 为安全英文文本 |
| 错误机器码 | `INVALID_REQUEST`(400)、`UNAUTHENTICATED`(401)、`FORBIDDEN`(403)、`CORS_ORIGIN_DENIED`(403)、`CORS_PREFLIGHT_DENIED`(403)、`NOT_FOUND`(404)、`METHOD_NOT_ALLOWED`(405)、`VERSION_CONFLICT`(409)、`STATE_CONFLICT`(409)、`IDEMPOTENCY_CONFLICT`(409)、`UNSUPPORTED_MEDIA_TYPE`(415)、`VALIDATION_FAILED`(422)、`INTERNAL_ERROR`(500)、`SERVICE_UNAVAILABLE`(503) |
| GET endpoint 的 HEAD | 阶段三不支持 HEAD；订单列表/详情 GET 路由必须显式返回 `405 METHOD_NOT_ALLOWED`，准确提供该 route 的 `Allow`，不得因 `http.ServeMux` 的 GET/HEAD 匹配行为意外返回成功 |

`INVALID_REQUEST` 专用于 transport/结构错误：JSON 语法、类型、未知字段、多余值、body 超限、缺失/非法幂等 header，以及非法 query 结构、枚举或范围。`UNSUPPORTED_MEDIA_TYPE` 专用于缺失、非法或非 JSON Content-Type。`VALIDATION_FAILED` 专用于 typed command 的业务字段语义：trim 后空字符串、字符串/明细/数量/金额边界、缺失或非 `CNY` 币种。认证、授权、存在性和冲突分别使用冻结的 401/403/404/409 语义；字段级错误 detail 使用请求中的 camelCase 字段路径。

阶段三扩展现有公共错误 envelope：`code`、`message`、`requestId` 保持不变，只在字段可定位的 `INVALID_REQUEST`/`VALIDATION_FAILED` 中可选增加 `details`。认证、授权、资源、冲突、内部和服务不可用错误不返回 `details`；阶段一/二客户端忽略新增可选字段即可保持兼容。

订单 HTTP 请求固定按以下顺序短路，组合错误不得改变优先级：已知路由/path pattern 与 method → CORS actual/preflight 检查 → authentication → endpoint role authorization → Content-Type → create 的 `Idempotency-Key` / query / orderId path 结构 → JSON 结构解码 → typed command validation → repository 资源存在性 → version → state。CORS middleware 使用显式 route allowlist 先判定已知 path/method，因此 unknown path 保持 404；对已知实际请求，非法 Origin 先于认证拒绝。只有通过前一层才执行后一层；例如非法 Origin + 缺失 token 返回 CORS 403，允许 Origin 下 viewer 的错误 Content-Type 写请求返回业务 `403`，operator 的非法 Content-Type + 损坏 JSON 返回 `415`，operator 的非法 orderId + 损坏 JSON 返回 `404`。

关键词 `q` 固定在规范化查询中匹配订单 ID、客户名、SKU 和明细名；trim 后空字符串等同未提供，`%`、`_` 和转义字符按字面量处理。阶段三冻结为 SQLite 可可靠支持的 ASCII 大小写不敏感：ASCII 字母先统一折叠后比较，中文等无大小写字符按原值比较；不声称完整 Unicode case folding，也不依赖默认 `NOCASE` 处理非 ASCII。`createdFrom > createdTo` 返回 `400 INVALID_REQUEST`。同一 query 参数出现多次一律拒绝，不采用首值或末值。

## 4. CORS 契约

阶段二延期的浏览器 CORS 在阶段三批次 E2 落地。API 配置新增可选 `CORS_ALLOWED_ORIGIN`：未设置时保持当前无 CORS 行为；设置时必须是单个绝对 `http`/`https` origin，仅含 scheme、host 和可选 port，不允许 `*`、userinfo、path、query 或 fragment。development/production 均不提供内置默认值，真实 Schema-UI 联调必须显式配置可信 origin。

- 无 `Origin` 的同源/非浏览器请求按原路由处理；匹配的实际跨源请求返回精确 `Access-Control-Allow-Origin` 和 `Vary: Origin`，不启用 credentials。
- 预检 `OPTIONS` 位于 authentication 之前，仅对匹配 origin、目标 method 和请求 header 成功返回 `204`；允许 method 为 `GET, POST, PATCH, OPTIONS`，允许 header 为 `Authorization, Content-Type, Idempotency-Key, X-Request-ID`，`Access-Control-Max-Age: 600`。
- 预检成功和拒绝结果都返回与缓存判断因素一致的 `Vary: Origin, Access-Control-Request-Method, Access-Control-Request-Headers`；实现必须追加或合并 `Vary`，不得覆盖其他 middleware 已写入的值。实际响应至少保留 `Vary: Origin`。
- 实际请求与预检的 origin 不匹配、origin 非法、请求 method/header 不在 allowlist 时返回 `403 CORS_ORIGIN_DENIED` 或 `403 CORS_PREFLIGHT_DENIED`，复用 request ID error envelope；不返回 `Access-Control-Allow-Origin`。
- 响应公开 `X-Request-ID`，即 `Access-Control-Expose-Headers: X-Request-ID`。CORS 只控制浏览器 transport，不替代 authentication、authorization 或业务校验。

完整 middleware 顺序固定为 request ID → access log → recovery → CORS → authentication → authorization → route handler，与阶段二已冻结的前三层保持一致。CORS 拒绝和预检均已处于 request ID、access log、recovery 内，因此返回 `X-Request-ID` 并进入访问日志。无 `Origin` 请求完全跳过 CORS；非 OPTIONS 的非法/不匹配 Origin 在 authentication 前拒绝。`OPTIONS` 只对显式 CORS route allowlist 中的已知 API path 处理预检，未知 path 返回普通 `404` 且不返回 CORS allow header。

配置装配固定为 `config.LoadAPI` 校验 `CORS_ALLOWED_ORIGIN` → `config.APIConfig.CORSAllowedOrigin` → `app.NewAuthenticatedAPIWithDependencies` → `httpapi.Dependencies.CORSAllowedOrigin`。非法 origin 必须使 `config.LoadAPI` 失败，进程不得继续到 app 构造、监听端口、创建/打开数据库或执行 migration。配置测试覆盖 development/production 的未设置、合法 http/https、通配符、多个 origin、userinfo、path/query/fragment；未设置始终合法且表示禁用，不因环境隐式启用。

应用装配边界固定为：只有 `app.NewAuthenticatedAPI`/`NewAuthenticatedAPIWithDependencies` 打开共享 `store.DB`、创建 auth service、order repository 和 order service，并将非 nil `OrderService` 与 `CORSAllowedOrigin` 放入 `httpapi.Dependencies` 后注册订单路由；任一步失败都使构造器返回错误且不启动 HTTP server。`app.NewAPI` 继续只提供健康检查、readiness 和 fallback，不打开业务数据库、不注册认证或订单路由；`OrderService == nil` 时 `httpapi.NewHandler` 不注册订单路由，不得留下会 panic 的半注册 handler。测试必须证明基础 API 的订单 path 为 404，认证 API 才注册订单路由且仍经过 auth middleware。

生命周期顺序固定为：先停止接收请求并完成 `http.Server.Shutdown`，再关闭 readiness probe，使 order/auth service 不再可达，随后关闭二者共享且由 app 拥有的 `store.DB`，最后释放进程锁。order/auth service 不拥有数据库且阶段三不单独引入无意义的 `Close`；若未来增加自有可关闭资源，必须在共享 DB 之前关闭并增加顺序测试。`httpapi.NewHandler` 只注册 route/middleware，不创建数据库或 service；订单 handler 通过 `PrincipalFromContext` 获取主体，order service 接收 Principal 并执行业务授权。共享 `routeMetadata` allowlist 同时驱动 known path、mux 注册、支持 method、OPTIONS 预检 method/header 和 method mismatch 的 `Allow`，测试证明三者不会漂移。

## 5. HTTP 端点

从目标文档迁入当前文档的接口为：

- `GET /api/v1/orders`：分页列表；
- `POST /api/v1/orders`：创建草稿；
- `GET /api/v1/orders/{orderId}`：详情；
- `PATCH /api/v1/orders/{orderId}`：编辑草稿；
- `POST /api/v1/orders/{orderId}/confirm`；
- `POST /api/v1/orders/{orderId}/fulfill`；
- `POST /api/v1/orders/{orderId}/ship`；
- `POST /api/v1/orders/{orderId}/complete`；
- `POST /api/v1/orders/{orderId}/cancel`。

上述常量是批次 A 的实现输入，不得延后到 handler 实现时临时决定。排序字段必须映射到固定 SQL 片段，不得把 query 直接拼接入 SQL；实现若必须改变冻结值，应先修改本计划、checklist 和目标 API，再修改代码与测试。

## 6. Seed 基线

`order_demo=1` 使用符合生产格式的固定 hex seed ID，并冻结以下最小组合；固定 ID 只用于 development seed，展示含义由 `customerName` 表达，生产创建仍使用随机 ID：

| 固定订单 ID | 履约状态 | 支付状态 | 后续用途 |
|---|---|---|---|
| `ord_00000000000000000000000000000001` | `DRAFT` | `UNPAID` | 编辑、确认、取消 |
| `ord_00000000000000000000000000000002` | `CONFIRMED` | `UNPAID` | 开始履约、取消 |
| `ord_00000000000000000000000000000003` | `FULFILLING` | `PAID` | 发货、取消、阶段四退款前置 |
| `ord_00000000000000000000000000000004` | `SHIPPED` | `PAID` | 完成、阶段四退款前置 |
| `ord_00000000000000000000000000000005` | `COMPLETED` | `PAID` | 阶段四部分/全额退款前置 |
| `ord_00000000000000000000000000000006` | `CANCELLED` | `UNPAID` | 终态冲突 |

上述组合只约束阶段三 HTTP 写路径和 `order_demo=1` seed，不作为 orders 表的交叉 CHECK，也不使 repository 读取其他“两个枚举各自合法”的组合失败。数据库只分别约束订单状态与支付状态枚举；因此 `DRAFT + PAID` 不视为 schema 损坏，但阶段三 HTTP/seed 不产生它。履约 Action 不改变支付状态。阶段四扩展写路径或 seed 组合时必须先更新领域模型与本契约。

六个 seed 订单的 `created_at` 固定为 `2026-01-01T00:00:00Z` 至 `2026-01-01T05:00:00Z`，按表中顺序每条递增一小时；每条 `updated_at == created_at`。`SeedOrderDemo(ctx, appliedAt)` 接收注入时间，仅用于新写入的 `seed_versions.applied_at`，生产默认传 `time.Now().UTC()`，测试传固定值；已是 version 1 的只读一致性校验不更新时间。排序与场景断言只依赖订单固定时间，不依赖 seed 元数据时间。

admin 的 development `seedDemo`/`resetDemo` 固定按 `runtime → auth_demo → order_demo` 调用；`SeedOrderDemo` 只接收 `appliedAt`，admin production path 传 `time.Now().UTC()`，测试直接传固定时间。每个 group 的结果和 seed_versions 行都可断言顺序；重复 seed 只读校验且不更新时间。

## 7. 代码边界

```text
internal/
  order/                 # 状态机、金额、capability、OrderService 与窄 repository 接口
  store/                 # migration v3、订单/明细/幂等 repository、订单 seed
  httpapi/               # orders 路由、严格 JSON/query 解码、认证和错误映射
  app/                   # 订单 service 与 repository 的依赖装配
```

包名固定为单数 `internal/order`，与 `internal/auth` 的领域包风格一致。`internal/order` 不依赖 HTTP 和 SQL，并定义 `OrderService` 及其所需的窄 repository 接口；`internal/store` 只实现参数化查询、事务和数据库约束；`internal/httpapi` 只做 transport、认证主体获取和错误映射，不直接拼 SQL、不直接执行状态转换。`OrderService` 是状态、权限、version、幂等和金额不变量的唯一协调者。

构造器边界同样属于代码架构：`app.NewAPI` 是不带业务认证依赖的基础模式，不装配 order package；`app.NewAuthenticatedAPIWithDependencies` 是订单 API 的唯一应用入口，必须先成功创建共享 DB、auth service、order repository/service，再构造 handler。`httpapi.NewHandler` 以非 nil `OrderService` 作为订单 route registration 的前提，不能注册后在请求期才因 nil 依赖返回 panic 或临时 500。

repository 接口推荐使用以下业务类型形态，不暴露 `*sql.DB`、`*sql.Tx` 或 SQL 字符串；方法名和参数组织允许等价调整，但必须保留窄接口、repository transaction ownership 和 service 不接触 `database/sql` 的边界：

```go
type OrderRepository interface {
  ListOrders(context.Context, ListQuery) (Page, error)
  GetOrder(context.Context, string) (Order, error)
  CreateOrder(context.Context, CreatePersistence) (CreateResult, error)
  UpdateDraft(context.Context, UpdateDraftPersistence) (Order, error)
  TransitionOrder(context.Context, TransitionPersistence) (Order, error)
  ReplayCreate(context.Context, ReplayKey) (CreateSnapshot, error)
}
```

transaction ownership 固定属于 repository：ListOrders 内部建立显式只读事务完成 bounded COUNT + page；CreateOrder/UpdateDraft/TransitionOrder 内部建立写事务并提交领域结果。`OrderService` 编排权限、金额、version、状态和幂等 command，但不接触 `database/sql`。列表必须证明固定的 COUNT + page 有界查询计划；通常为两条 SQL，也允许等价 executor/query 组织，只要没有逐订单 item 查询、逐行事务或第三条补查询。两条查询均可使用参数化 `EXISTS`/受控 join 访问 `order_items` 以支持 `q` 对 SKU/name 的筛选。列表 capability 只由 principal 与订单 status 计算，page 返回订单标量；详情才返回有序 items。

冻结的是外部契约和可观察边界：endpoint、角色矩阵、状态转换、金额范围、DTO、错误码、幂等语义、分页排序、CORS header/短路顺序、事务 snapshot 一致性、查询有界性和取消/日志结果。以下属于推荐实现而非不可替换 API：repository 方法名、`applyMigration` helper 名称、具体 executor/barrier seam、测试函数名，以及使用 `http.ErrAbortHandler` 的具体连接中止机制。实现可以采用等价方案，但必须提供相同的事务一致性、查询上限、连接中止、错误分类和资源清理证据。

错误分类矩阵：

| 错误 | 拥有/产生层 | service 处理 | HTTP 映射 |
|---|---|---|---|
| `context.Canceled` / `context.DeadlineExceeded` | 任意 I/O 层 | 原样保留，保持 `errors.Is` | 若尚未提交响应则通过 `http.ErrAbortHandler` 或等价机制中止连接，不映射 503，不写业务 envelope；access log 记为 canceled outcome |
| `NotFound` | store/order repository | 原样保留 | `404 NOT_FOUND` |
| `VersionConflict` | order service/repository 条件更新分类 | 原样保留 | `409 VERSION_CONFLICT` |
| `StateConflict` | order service | 原样保留 | `409 STATE_CONFLICT` |
| `IdempotencyConflict` | order service 比较 digest | 原样保留 | `409 IDEMPOTENCY_CONFLICT` |
| `Unavailable` | store 仅从 SQLite BUSY/LOCKED 转换 | 原样保留 | `503 SERVICE_UNAVAILABLE` + `Retry-After: 1` |
| `Validation` | order typed command 校验 | 携带安全 field details | `422 VALIDATION_FAILED` |
| 损坏行/snapshot、关闭 DB、其他 SQL/scan | store | 包装为 typed `Internal`，保留 cause 供日志但不对外 | `500 INTERNAL_ERROR` |

错误类型支持 `errors.Is`/`errors.As`；store 通过驱动错误码分类 BUSY/LOCKED，不匹配错误字符串。HTTP 只对 exported/typed 分类做映射，不读取错误文本决定状态码。

## 9. 非目标与风险

- 不实现退款创建/审批/拒绝；阶段三只保证已支付 seed 和可退款前置数据。
- 不实现附件上传、绑定、下载、看板和页面配置。
- 不加入库存、物流、支付网关、多租户或账号管理。
- 每个 `store.DB` 当前固定 `SetMaxOpenConns(1)`：同一 API 进程的第二个事务先等待 database/sql 连接池，不会进入 SQLite busy handler。该路径只验证等待期间尊重 context deadline/cancel，取消后不写入第二个订单。
- SQLite busy/locked 必须用两个独立 `store.DB` 打开同一临时数据库并持有首个写事务来稳定触发；生产 `busy_timeout` 保持 5000ms，测试通过可注入连接选项或等价 seam 使用不超过 500ms 的短 timeout，避免每个错误路径真实等待 5 秒。超时后 repository 返回类型化 unavailable 错误；HTTP handler 使用注入的 order service/repository stub 验证该错误映射为 `503 SERVICE_UNAVAILABLE` 与 `Retry-After: 1`，不要求真实 HTTP 在单连接池内触发 busy。
- 两个独立连接竞争相同 key 时最多创建一个订单；竞争请求允许在首事务及时提交后直接读取并重放 snapshot v1，也允许因 busy/locked 返回 `503`。若返回 `503`，使用同一 key 重试后必须重放首次创建快照；任何路径都不得产生第二个订单。本阶段不扩大连接池，也不增加跨进程“进行中”状态表。
- 单 DB 连接池等待测试与双 DB SQLite BUSY/LOCKED 测试严格分离：前者只断言 channel/barrier 协调下的 context cancel/deadline 原样保留，不映射 503；后者只通过 modernc SQLite 结构化错误类型/错误码识别 BUSY/LOCKED，转换为 typed `Unavailable`，禁止按错误字符串或精确 sleep 判定。
- handler 尚未写响应时收到 `context.Canceled`/`context.DeadlineExceeded`：通过 `http.ErrAbortHandler` 或等价连接中止机制中止连接，不写业务 error envelope、不映射 503；recovery middleware 不得把该取消路径映射为 `500 INTERNAL_ERROR`。access log 使用独立的 canceled outcome，不把未提交响应记为 200 或业务失败；保留 `errors.Is`。已提交响应后的取消不尝试改写 status 或 body。测试必须使用真实 `httptest.Server` 或连接级客户端观察连接中止，不能只断言 `httptest.ResponseRecorder`。
- snapshot v1 是创建时不可变持久化 DTO，不是当前订单缓存；后续编辑/Action 不更新它。读取未知版本、损坏快照或元数据错配时安全失败，不回退为当前订单查询或重新创建。
- 阶段三不提供在线幂等记录清理，development reset 继续通过重建数据库清除。进入阶段四或生产化前必须另行冻结按 endpoint/operation 的保留期、归档或清理策略，以及 snapshot 版本兼容迁移规则；在此之前不得后台静默删除或原地重写 snapshot v1。
- 所有连接池等待、双 DB BUSY/LOCKED、并发幂等、并发 Action 和真实连接中止测试都使用 `context.WithTimeout` 或测试级 deadline，单测试上限为 10 秒；cleanup 必须释放 barrier/channel、回滚未提交事务、关闭所有 DB/server 并删除临时目录。超时必须使测试失败并完成清理，不得让 CI 无限等待。
- 搜索不承诺完整 Unicode case folding；测试必须包含 ASCII 大小写和中文原值 fixture，避免把 SQLite 默认 `NOCASE` 误描述为 Unicode 能力。

repository/service 错误分类固定如下：幂等唯一键冲突必须重新读取记录并按 digest 分类为 replay 或 `IDEMPOTENCY_CONFLICT`；只有驱动分类为 SQLite BUSY/LOCKED 才转换为内部 typed `Unavailable`；`context.Canceled`/`context.DeadlineExceeded` 原样保留，不转换为 503；关闭数据库、其他 SQL/扫描错误、损坏数据和 snapshot 错配转换为 typed internal error。handler 只把 `Unavailable` 映射为 `503` + `Retry-After: 1`，普通事务失败不得自动重试创建。

JSON 数字解码固定使用 DTO 的 `json.Number` 配合严格单值 decoder，而不是先解码 float64。quantity、unitPrice、version 只接受十进制整数词法 `^-?(0|[1-9][0-9]*)$`；`1.0`、`1e2`、字符串、null 和超出 int64 均为 `400 INVALID_REQUEST`，解析成功后的范围错误才进入 `422 VALIDATION_FAILED`。

查询时间在 HTTP 边界使用 `time.Parse(time.RFC3339)`，例如 `2026-01-01T00:00:00+08:00` 规范化为 `2025-12-31T16:00:00Z`；repository 接收 `time.Time` 并按时间瞬间比较，响应和数据库订单时间始终输出 UTC `Z`，不得直接按原始 offset 字符串比较。

具名自动化证据固定为：`TestListOrdersUsesBoundedQueries` 通过测试 executor 计数证明列表固定执行 COUNT + page 两条语句且无逐行查询；`TestListOrdersUsesSingleSnapshot` 或等价测试使用协调点与第二个 DB，证明 COUNT 与 page 来自同一 snapshot；`TestOrderRepositoryRejectsCorruptRows` 通过测试 SQL 注入非法存量值覆盖扫描/数据损坏；`TestOrderRepositoryConnectionWaitHonorsContext` 覆盖单连接池等待；`TestOrderRepositoryClassifiesBusyAcrossConnections` 覆盖双 DB BUSY/LOCKED；`TestOrderHTTPMapsUnavailable` 覆盖 503。文档门禁使用 `docs/audit/validate.ps1` 检查 frontmatter、相对链接与 `git diff HEAD --check`，`validate.tests.ps1` 固定验证合法 fixture 成功、缺失链接 fixture 失败和 fragment 只校验文件目标。测试名称若因 Go 命名调整，必须在 checklist 证据中记录一一对应的实际名称，不能只写“代码审查通过”。

## 10. 完成标准

阶段三硬门禁：

- 九个订单 endpoint 具备当前 API 文档、真实 HTTP 集成测试和稳定错误契约；
- 查询、金额、状态机、权限、version 和幂等均有 repository/service 测试；
- 全部合法履约转换和主要非法转换有证据，viewer/approver 的写权限被后端拒绝；
- seed 可重复并覆盖阶段四所需订单/支付状态；
- 认证应用完成订单装配，基础 API 不暴露订单路由；CORS 配置失败无启动副作用；
- 最小跨源 OPTIONS + 带认证订单请求 smoke 通过，但不宣称页面联调完成；
- 订单验证矩阵改为 `enabled: yes`，幂等验证矩阵改为 `enabled: yes`；
- 全量 test、vet、race 和文档检查通过后，阶段三 checklist 工程项全部勾选并提交完成报告。

阶段六门禁独立包含页面 YAML、L0-L4 校验、页面 action/reaction、完整 Schema-UI 浏览器场景和页面级回归证据；这些未完成不阻塞阶段三归档，也不得由阶段三最小跨源 smoke 冒充完成。

归档是完成报告之后的后置动作，不属于 checklist：先汇报已完成项、未执行项和剩余风险，只有用户明确确认后，才将 `0003` 计划/checklist 移入 `docs/audit/archived/` 并更新活跃与归档索引。
