## D. 密码、角色与 JWT 核心

- [x] D1. 密码 hash/compare 封装在 `internal/auth`，HTTP 和 store 不直接调用 bcrypt。
- [x] D2. bcrypt cost 固定并有测试；所有密码长度使用 UTF-8 编码后的 byte 数校验为 12..72，支持识别低 cost hash 供未来升级且登录时不静默改库。
- [x] D3. 未知用户名使用进程启动时预生成、与真实 hash 相同 bcrypt cost 的 dummy hash，不按请求重复生成；对外结果与错误密码一致，已知与未知用户在 compare 前使用相同的密码长度校验路径。
- [x] D4. username 固定使用 `strings.TrimSpace` + `strings.ToLower`，数据库查询与 limiter key 复用同一结果；密码不 trim、不做 Unicode 变换。
- [x] D5. `Role` 只允许 viewer/operator/approver/admin，授权使用显式 allowlist/capability。
- [x] D6. token 使用 HS256 和至少 32-byte key，TTL 固定 15 分钟。
- [x] D7. claims 包含并验证 `sub`、role、`exp`、`iat`、`jti`、issuer 和 audience；自定义 validation 强制 `sub`、`iat`、`jti` 和 role 非空且 role 合法。
- [x] D8. parser 显式使用 `WithValidMethods`、`WithExpirationRequired`、`WithIssuer`、`WithAudience`、`WithIssuedAt`、`WithTimeFunc` 和 30 秒 `WithLeeway`。
- [x] D9. 测试覆盖篡改签名、`alg=none`、错误 HMAC 算法、issuer/audience、claims 缺失和格式错误。
- [x] D10. 注入 clock 覆盖 30 秒临界 leeway：拒绝 `iat > now + 30s`，并在 `now >= exp + 30s` 时判定 JWT 过期；测试覆盖等号前一瞬与等号点且不使用 sleep。
- [x] D11. 测试分别删除 `sub`、`iat`、`jti` 或 role，证明 `WithIssuedAt` 的可选语义不会允许缺失 `iat`，且 parser 使用注入 clock 而非进程时间。
- [x] D12. ID 与 `jti` 使用注入的 `crypto/rand` 默认实现，生成失败向上传播且不创建 session。
- [x] D13. `go test ./internal/auth -count=1` 通过。

证据：`go test ./internal/auth -count=1` 通过；覆盖 byte 边界、dummy compare、随机源失败、严格 claims、`alg=none`、错误 HMAC 算法、格式错误、签发方、受众、篡改，以及 iat/expiry 等号边界。

## E. 用户、Session 与认证服务

- [x] E1. auth service 通过窄 store interface 查询用户、创建 session、认证 principal 和撤销 session。
- [x] E2. 登录成功创建唯一 session，JWT 的 `sub`/role/`jti` 与数据库一致，session `expires_at` 与 JWT `exp` 完全相同。
- [x] E3. 同一用户可并存多条未撤销 session，新登录不撤销既有 session。
- [x] E4. 用户不存在、密码错误或用户禁用均不创建 session。
- [x] E5. 每次认证同时检查 JWT、session 存在、未撤销、未到期、`session.user_id == claims.sub` 和用户未禁用。
- [x] E6. 每次认证读取当前用户角色；数据库角色与 token role 不一致时拒绝旧 token。
- [x] E7. session 到期以服务端 clock 与数据库值为准，在 `now >= expires_at + 30s` 时过期，与 JWT `exp` 边界完全一致。
- [x] E8. logout 使用条件更新只撤销当前 session，不撤销同一用户的其他 session。
- [x] E9. 两个在撤销前均通过 middleware 的并发 logout 可幂等返回 `204`；撤销提交后新到达的旧 token 请求返回 `401`，第二次条件更新未命中不得变成 `500`。
- [x] E10. store 关闭、context 取消、事务失败和随机源失败均返回类型化内部错误且不泄露细节。
- [x] E11. repository/service 测试覆盖重复 `jti`、多 session、session 缺失、subject/session 错配、撤销、到期、禁用和角色变化；用户 A 的合法 `jti` 搭配用户 B 的 `sub` 必须被拒绝。
- [x] E12. `go test ./internal/auth ./internal/store -count=1` 通过。
- [x] E13. 已记录本阶段不清理过期/撤销 session：development 依赖 reset，production 会持续增长且不得宣称生产完备；索引为后续 purge 保留基础。

证据：`go test ./internal/auth ./internal/store -count=1` 通过；覆盖重复 jti、多 session、session 缺失、subject/session 错配、撤销、到期、禁用、角色变化、幂等撤销、真实 store 关闭/context 取消、随机源与 session 写入失败；内部错误可分类且错误文本脱敏。

## F. 登录限流与输入边界

- [x] F1. login 只接受 JSON，body 上限固定为 4 KiB，并拒绝未知字段和多余 JSON 值。
- [x] F2. username/password 缺失、类型错误、空值和超长输入返回安全 `INVALID_REQUEST`；覆盖 ASCII 与多字节 Unicode 的 11/12/72/73-byte 边界，不按 rune 数判断。
- [x] F3. limiter 从 `RemoteAddr` 用 `net.SplitHostPort` + `net/netip` 提取规范 IP；同 IP 不同端口得到同一 key，覆盖 IPv4、IPv6、malformed address 与固定 `unknown` 分区，且不信任代理 header。
- [x] F4. 每个 key 在 1 分钟内最多 5 次失败；第 6 次及后续在密码校验前返回 `429 RATE_LIMITED`，正确密码也不能绕过。
- [x] F5. 成功登录清除当前 key 的失败计数，不影响其他用户名或 IP。
- [x] F6. limiter clock 可注入，窗口边界测试不使用 sleep。
- [x] F7. `Retry-After` 为剩余窗口秒数向上取整且最小为 `1`，窗口结束后恢复认证。
- [x] F8. 只有未知用户、错误密码和禁用账号增加计数；`400`、`415`、context 取消/超时和内部错误不计数，`429` 不增加计数或延长窗口。
- [x] F9. limiter 最多保存 4096 个活跃 bucket且每个规范 IP 最多占用 32 个 username bucket；`unknown` 是独立 IP 分区。
- [x] F10. 单 IP 达到 32 个 key 后，新 username 返回 `429` 与 `Retry-After: 1`，不插入 bucket且不影响其他 IP。
- [x] F11. 全局满载时只淘汰最久未访问且失败数小于 5 的未封禁 bucket；窗口内失败数达到 5 的受限 bucket不可淘汰，全部受保护时新 key fail-closed。
- [x] F12. 单 IP 大量用户名与多 IP 高基数测试证明 bucket 总数不超过 4096，单 IP 不能耗尽全局容量或使其他 IP 的已有/新登录被全局拒绝。
- [x] F13. 过期 bucket 被清理，窗口结束后可为新 key 释放容量。
- [x] F14. 并发失败请求、每 IP 配额与并发满载计数正确且通过 race detector。
- [x] F15. `go test ./internal/httpapi ./internal/auth -count=1` 通过。

证据：`go test ./internal/httpapi ./internal/auth -count=1` 与 `go test -race ./...` 通过；覆盖字段/类型/ASCII 与 Unicode byte 边界、成功清计数、400/415/cancel/timeout/internal 分类、代理 header 忽略、固定窗口、32/4096 容量、单 IP 洪泛隔离、保护/清理和并发失败。
