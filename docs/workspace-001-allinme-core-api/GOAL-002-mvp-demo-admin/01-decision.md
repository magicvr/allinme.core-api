# 决策记录 · GOAL-002

## 信息需求与阶段门禁

权威表见 [00-meta.md](00-meta.md)。

- **I-001**（策略）：已裁决 → **A 协议演进**（`decided`）。
- **I-008**（新协议制品）：`collecting`；**关闭前不得解除 blocked、不得冻结基于新协议的 MVP 方案、不得实施**。
- I-002～I-005：仍为 required，但**暂停主动收集**，待 I-008 关闭后按新协议边界再推进，避免按 2.0.0 缺口返工。

## D-001 · MVP 范围与验收口径

**决定**：

1. 模块范围见 meta 模块表（真实登录/会话、真实用户角色权限、schema 菜单路由、订单/钱包/通知、仪表盘 grid、行内动作、批量动作；上传不做）。
2. 验收以**本服务 page schema 覆盖 Admin 全部入口**为准；不强制本目标关门前完成外部 Renderer 浏览器联调。
3. 实施所钉协议版本：**不再以 2.0.0 为最终实施基线**；待含批量等能力的新协议发布后，由 Root 决策重新钉死（见 Root D-005 / 本目标 D-005）。2.0.0 仅作当前对照与缺口依据。

**为什么**：

- 用户明确 demo 完整 Admin 与三域通用业务，便于证明复用价值。
- 本仓仅后端；schema 覆盖入口是可独立验证的后端交付物。
- 策略 A 要求实施落在**新协议**上，继续钉 2.0.0 会与批量需求冲突。

**未选方案**：

- **登录/权限用纯 mock**：与「真实」要求不符。
- **验收必须浏览器 E2E**：依赖外部仓，阻塞本仓后端交付节奏。

## D-002 · 批量动作策略（已裁决）

**已核实事实**（对照 `schema-ui-docs` 2.0.0）：

| 能力 | Schema-UI 2.0 核心协议 |
|------|------------------------|
| 行内动作（`RowAction`，含 `actionRef` 声明式后端请求） | **支持** |
| 批量操作 | **不支持**（`docs/decisions/0019-v2-admin-scope.md`） |
| 标准详情/编辑导航、表单 initialValues 回填等 | **不在** v2.0 核心保证内 |

**决定（2026-07-23 · 用户确认）**：

- 行内动作：纳入 MVP，按**目标协议**声明式能力实现。
- 批量动作：纳入 MVP，且选择 **A. 推动协议演进**——在 `schema-ui-docs` 为批量等能力立 ADR + Schema + fixtures，**发布新协议版本后再在本仓实现**。
- **明确不选** B（Host Extension）与 C（MVP 缩 scope 去掉批量）。

**为什么**：

- 批量需成为可跨项目互操作的协议能力，而非本仓私有扩展。
- 用户接受本项目因此暂停，优先协议完整性。

**未选方案**：

- **B. Host Extension**：交付快但无跨 Renderer 协议保证。
- **C. 缩 scope**：放弃 MVP 批量，与「需要批量」不符。
- **假装 2.0.0 已支持批量**：禁止。

## D-003 · 业务域选型

**决定**：MVP 业务域为 **订单、钱包、通知** 三套；具体字段与状态机待 I-008 关闭后继续（I-003）。

**为什么**：用户指定。

## D-004 · 非目标（本 MVP）

**决定**：不实现上传；不将生产级多租户、计费、复杂工作流引擎纳入本目标。

**为什么**：用户明确上传暂缓；保持 demo 完整与可关门范围。

## D-005 · 本仓暂停直至新协议完成

**决定**：

1. `GOAL-002-mvp-demo-admin` 与父目标实施路径 **`status: blocked`**。
2. **暂停**本仓针对 MVP 的方案细化与代码实施（含鉴权/领域/page 生产等），避免建立在将被替代的 2.0.0 缺口假设上。
3. **恢复条件（须同时满足）**：
   - `schema-ui-docs` 已发布含 MVP 所需能力（至少：**批量动作**核心协议支持；并尽量覆盖 I-005 所涉 CRUD 生命周期缺口，或书面说明仍不支持项与 residual）的新协议制品；
   - I-008 在 meta 中更新为 `verified`，并记录版本号、tag 或 SHA-256；
   - Root 完成「重新钉死协议版本」决策（取代/修订原 2.0.0 钉死）；
   - 用户确认本仓恢复 `/govern` 推进。
4. 协议演进工作在 **`schema-ui-docs` 仓库**进行，**不在本仓**伪造协议语义。

**为什么**：用户明确「先推动协议 → 本项目暂时停止」。

**未选方案**：

- **本仓边做边等协议**：高返工风险。
- **本仓私自扩展 page 字段冒充协议**：违反权威层级。
