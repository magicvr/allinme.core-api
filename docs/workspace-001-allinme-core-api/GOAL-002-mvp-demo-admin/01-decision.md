## D-001 · MVP 范围与验收口径

**决定**：

1. 模块范围见 meta 模块表（真实登录/会话、真实用户角色权限、schema 菜单路由、订单/钱包/通知、仪表盘 grid、行内动作、批量动作；上传不做）。
2. 验收以**本服务 page schema 覆盖 Admin 全部入口**为准；不强制本目标关门前完成外部 Renderer 浏览器联调。
3. 实施协议版本：以 Root **D-006** 钉死的 **2.4.1** 为准（本目标 D-006 确认采纳）。2.0.0 仅历史对照。

**为什么**：

- 用户明确 demo 完整 Admin 与三域通用业务，便于证明复用价值。
- 本仓仅后端；schema 覆盖入口是可独立验证的后端交付物。
- 策略 A 要求实施落在含批量的协议上。

**未选方案**：

- **登录/权限用纯 mock**：与「真实」要求不符。
- **验收必须浏览器 E2E**：依赖外部仓，阻塞本仓后端交付节奏。

## D-002 · 批量动作策略（已裁决）

**已核实事实（历史）**（对照 `schema-ui-docs` 2.0.0）：

| 能力 | Schema-UI 2.0 核心协议 |
|------|------------------------|
| 行内动作 | **支持** |
| 批量操作 | **不支持** |
| 标准详情/编辑导航、表单回填等 | **不在** v2.0 核心保证内 |

**决定（2026-07-23 · 用户确认）**：

- 行内动作：纳入 MVP，按目标协议声明式能力实现。
- 批量动作：纳入 MVP，策略 **A. 推动协议演进** 后实现。
- **明确不选** B（Host Extension）与 C（MVP 缩 scope 去掉批量）。

**2026-07-24 落地**：协议侧已在 2.4.1 提供 `table.selection` + `actions.batch.request`（自 2.2）；本仓实现须对照钉死制品，禁止 Host Extension 冒充协议。

## D-003 · 业务域选型

**决定**：MVP 业务域为 **订单、钱包、通知** 三套；具体字段与状态机继续由 I-003 推进。

**为什么**：用户指定。

## D-004 · 非目标（本 MVP）

**决定**：不实现上传；不将生产级多租户、计费、复杂工作流引擎纳入本目标。

另：**不要求**协议侧跨页全选、批量部分成功、行内单元格编辑、导入导出向导、异步任务、树表——这些保持协议后续轨 / residual，不在本 MVP 伪造成支持。

**为什么**：用户明确上传暂缓；保持 demo 完整与可关门范围。
