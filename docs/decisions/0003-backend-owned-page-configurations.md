# 后端维护页面 YAML 并通过 JSON API 下发

## 背景

demo 需要把 Schema-UI 页面、真实 datasource 和 Action 作为完整交付物展示。若页面只位于前端仓，后端无法保证演示页面引用的 API 与当前版本同步；若页面存数据库并动态编辑，首版会引入额外管理、审计和运行时校验复杂度。

## 决策

- 页面配置以 `internal/pages/yaml/*.yaml` 保存，和业务 API、测试及变更记录共同演进。
- `internal/pages` 使用包内 `//go:embed yaml/*.yaml` 将页面纳入二进制；Go 测试按固定 `schema-ui-docs` 版本校验，启动时执行防御性复验。
- 任一页面解析或校验失败时 API 拒绝启动，readiness 不得报告成功。
- 运行时使用显式 page ID allowlist 返回 JSON，并支持 ETag；请求路径不能直接映射文件系统路径。
- 页面可按已认证角色控制入口，但所有业务 API 仍独立执行认证、授权和状态校验。
- 本仓不复制 Schema-UI 字段表或解释协议语义，页面源文件只作为协议实例。

## 备选方案

- 前端仓维护页面：前端所有权清晰，但完整 demo 的页面/API 版本容易漂移。
- 数据库动态页面：支持运行时编辑，但需要管理 UI、版本、审计和回滚机制。
- 仅维护 JSON：运行时直接，但人工审查和编辑体验弱于 YAML。

## 后果

后端发布物可独立提供与 API 同版本的完整页面配置，前端只需实现协议 Renderer。每次页面变化必须通过协议校验和 API 场景测试；未来若引入动态配置，需要新的 ADR 定义发布、审批、缓存和回滚流程。
