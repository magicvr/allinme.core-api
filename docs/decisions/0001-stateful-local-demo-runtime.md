# 使用 SQLite 与本地文件构建有状态 demo

## 背景

本仓需要为前端提供合理数据和真实状态变化，同时保持单机启动、可重复演示和低运维成本。纯内存数据无法验证重启持久化、事务和并发；外部数据库与对象存储会提高首次使用门槛。

## 决策

- 结构化数据和文件元数据存储在 SQLite，应用负责 migrations、seed 和仅开发模式可用的 reset。
- 附件内容存储在应用受控的本地数据目录，数据库保存服务端文件名、摘要、大小、类型和绑定关系。
- 自动化测试使用独立临时数据库和文件目录；演示重启保留状态，reset/seed 可恢复固定基线。
- SQLite 启用 foreign keys、busy timeout 和 WAL；跨实体状态变化使用显式事务和乐观锁。

## 备选方案

- 进程内状态：启动最简单，但不能证明持久化、事务和真实操作结果。
- SQLite BLOB：事务边界集中，但大文件增加数据库备份和 I/O 压力。
- PostgreSQL + 对象存储：更接近生产基础设施，但超出 demo 初始化目标。

## 后果

单进程 demo 可以零外部依赖运行并保留真实操作结果。部署为多副本或迁移到生产基础设施时，需要替换 store/files 适配器并重新评估 SQLite 写并发和本地磁盘所有权。
