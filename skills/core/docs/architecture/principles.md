# 治理原则

本文件记录 Goal Governance 核心方法论的**元规则**（跨目标、跨交付形态长期适用的原则）。具体操作细则见根目录 [AGENTS.md](../../AGENTS.md)、[docs/README.md](../README.md) 与模板 [docs/templates/](../templates/)。Skills 编排落地见 `skills/prompts/`（`/govern`、`/audit`）。Skills 与其他消费适配器不改变这些规则的权威性。

工作区与共享资料固定引用的操作协议见 [workspace-protocol.md](workspace-protocol.md)。它把 P-001 的路线图/串行阶段、P-002 的证据意识与 P-005 的信息门禁应用到跨工作区边界，不新增第二套生命周期或状态真相；目标状态只在各自工作区根内保存。

| 编号 | 标题 | 一句话 |
|------|------|--------|
| [P-001](#p-001--目标可执行性与路线图) | 目标可执行性与路线图 | 大目标先路线图，再按阶段立项 |
| [P-002](#p-002--治理闭环与阶段质量意识) | 治理闭环与阶段质量意识 | 设立→方案→实施→关门，关键节点可审视与修正 |
| [P-003](#p-003--交叉审计与意见响应) | 交叉审计与意见响应 | 独立审出意见；编排器汇总并响应全部意见 |
| [P-004](#p-004--用户裁决点) | 用户裁决点 | 是否自审、意见冲突 → 问用户并给建议，不自动裁 |
| [P-005](#p-005--信息就绪与未知项门禁) | 信息就绪与未知项门禁 | 允许带未知立项；按阶段收集、验证或接受残余风险 |

## P-001 · 目标可执行性与路线图

**原则**：如果一个目标还不能直接执行（即存在明显的拆解需要），则必须先为其构建可追踪的高层路线图，明确主要阶段与先后关系，再进入具体子目标的创建与执行。

**记录位置**：路线图应写在该目标的 `00-meta.md` 或 `01-decision.md` 中，并随进展更新。

**价值**：

- 避免在范围未厘清时过早创建大量细粒度子目标，减少返工与编号噪音。
- 用可追踪的阶段与先后关系对齐协作者与 AI 的预期，降低「先干再说」的漂移。
- 为后续拆分子目标提供明确边界：先路线图，再立项，再执行。

**与实践的关系**：

- Root Goal（GOAL-001）的高层路线图即本原则的落地示例。
- 已可直接执行的小目标无需强行补路线图；「明显需要拆解」时才强制适用。
- 操作细则以 [AGENTS.md](../../AGENTS.md) 第 6 节为准。

## P-002 · 治理闭环与阶段质量意识

**原则**：目标治理不仅是「写文档推进」，而是带质量意识的闭环。目标态顺序为：

```text
设立目标 → 识别未知与信息就绪（P-005）→ 审视目标质量 → 建立方案与实施计划 → 审视方案/计划
  → 按计划实施并记录事实 → 审视实施事实（可含整改环）→ 关门审计后结项
```

**比例裁剪**：小范围、已可直接执行的目标可合并审视步骤（例如目标与方案一次审视），**不得**因此省略「有证据的事实」与「关门前结论」。大目标 / 高风险目标宜保留更完整的门禁意识。

**修正与整改**：

- 目标或方案审视不通过 → 修正后可再审视，再进入下一步。
- 实施事实审视不通过 → 提出整改方案与计划 →（宜再审视）→ 整改实施 → 再审视事实，直至可接受再进入关门路径。
- **实施事实 ≠ 实施记录**：审计应核对可验证证据（产物路径、可勾选成功标准等），而非仅流水账修辞。

**价值**：降低设立时目标偏移、方案不可行、虚标进度与草率关门。

**延后（非本原则强制）**：完整自动状态机、硬编码全部门禁枚举——由 Skills 编排提示词逐步落地。

## P-003 · 交叉审计与意见响应

**原则**：

1. **交叉审计**：允许在编排推进流程**之外**进行独立审计，用于降低同一闭环内「自写自审自通过」的自证幻觉。独立审计默认**只形成审计意见**，**不**直接改目标的 `status` / `progress` 或方案正文。
2. **意见响应在编排器**：`/govern`（或等价编排路径）负责推进生命周期，并**汇总、评估、响应**与当前焦点相关的**全部**审计意见（自审 `self` + 一次或多次独立审 `independent`）。
3. **多意见并存**：允许同时存在多条审计意见；编排器维护开放意见集合（**必改项未关闭前不得假装放行或关门**），而不是只认最近一条。该门禁**不依赖**是否存在意见冲突：只要有未关闭 required/必改项，即阻断对应推进。
4. **独立复审**：修正后可再次请求交叉审计；响应闭环仍归编排器。

### 意见状态（最小约定）

| 概念 | 最小判定 |
|------|----------|
| **相关意见** | 同一目标 `03-audit.md` 中，scope 覆盖当前焦点（整体 / 阶段 / 门禁）的编号条目 |
| **开放必改** | 标为 required/必改、且尚无关闭证据的 finding |
| **已关闭** | 可核对的修正事实（路径、决策号等）+ 可选复审；口头声称不算 |
| **冲突** | 见 P-004（verdict 相反或必改项互否） |

字段级响应/关闭流程由 Skills 编排器与 `04` 原语细化。

### 落盘规则（权威位置）

| 规则 | 约定 |
|------|------|
| **权威落点** | 被审计目标的 `<workspace-root>/GOAL-…/03-audit.md`（**唯一**正式意见台账） |
| **编号** | 与自审共用序列：`A-001`、`A-002`…（按该文件内已有最大编号 +1） |
| **条目头（最小）** | `source: self \| independent`、日期、scope（审什么）、`verdict`（pass / conditional / fail）；建议含 auditor（工具/模型） |
| **正文** | 短报告直接写在该节；须含 findings 与（若 fail/conditional）必改/建议项 |
| **长文** | 全文可放同目标 `attachments/`（如 `attachments/audit-A-00N-independent.md`），但 **`03-audit.md` 必须有对应编号节**：摘要 + verdict + findings 要点 + **附件链接** |
| **禁止** | 仅聊天输出未落盘；仅有附件无 `03-audit` 索引节；另建全局审计库替代目标下 `03-audit`（当前阶段） |
| **写入方** | 交叉工具可直接追加 `03-audit`；或输出 Markdown 后由人/本侧代贴——仍标 `source: independent` |
| **编排器认账** | 推进/放行时扫描该目标 `03-audit.md` 中带编号的条目（含链接的附件详情） |

**来源标注**：`source: self | independent` 为强制最小字段。更细结构（findings ID 等）由审计原语细化。

**产品面**：默认主入口仍为编排器；独立审计为交叉专用入口（如 `/audit`），**不是**与历史四填表原语并列的主路径。

**价值**：出意见与改状态适度分离；交叉意见可多次介入而不分裂生命周期所有者；意见可被编排器稳定发现。

**边界**：单 AI / 同模型场景下的「独立」是意图与入口分离级的弱独立，不等于法定第三方鉴证。

## P-004 · 用户裁决点

**原则**：下列情形**不得**由编排器静默自动裁决；须向用户说明事实、**给出建议**，并等待确认；用户决策宜留痕（`01-decision` 或审计响应记录）。

### 4.1 有独立审计、尚无自审计

当推进或过门禁时发现**已有独立审计意见、尚无自审计**时：

- **询问用户**是否还需要做一次自审计；
- **不**自动跳过自审，也**不**在未询问时强制自审；
- 用户选择后：需要则先自审再统一响应；不需要则基于现有意见（含独立审）进入响应/评估。

**延后**：基于对象版本指纹、覆盖度等的**自动**判定「可否跳过自审」——明确不在当前强制范围。

### 4.2 多条审计意见冲突

当多条意见在 **verdict 或必改项**上明显冲突时（例如一方 pass、另一方 fail；或对同一必改项一要一否）：

1. **停止自动放行**；
2. 展示冲突摘要（来源、编号、结论、关键差异）；
3. **给出编排器建议**（附简短理由）；
4. **由用户决策**（采纳一侧 / 折中 / 驳回某侧并说明）；
5. 决策后写入留痕，再进入修正、复审或放行；
6. **未决策前**不因「已有审计」进入下一阶段或关门。

**不视为冲突**：多条同向意见、可叠加的必改项——合并为开放意见集合一并响应即可。拿不准时宁可当冲突询问用户。

**价值**：最终责任在人；建议降低决策成本；避免静默取「更乐观」一侧。

## P-005 · 信息就绪与未知项门禁

**原则**：目标设立时不假定协作者已经知悉全部必需信息。目标可以带着已识别的未知、假设或待验证事实进入治理；但这些信息必须显式登记、按阶段设置最晚需要时间，并以证据、决策或用户接受的残余风险闭环。

### 信息需求登记

每个有已识别信息缺口的目标，在 `00-meta.md` 或 `01-decision.md` 维护信息需求表。最小列为：

| 字段 | 含义 |
|------|------|
| ID | `I-001` 起递增，便于在决策、执行与审计中引用 |
| 所需信息 / 问题 | 尚需知道或验证什么；假设应明确写为假设 |
| 级别 | `required`（在最晚需要阶段前阻断受影响门禁）或 `non-blocking`（继续跟踪但不自动阻断） |
| 影响门禁 | 影响目标定义、方案、实施、验收或关门的哪一项 |
| 最晚需要阶段 | 在哪个阶段前必须处理；可标为暂不阻断 |
| 验证 / 收集动作 | 来源、实验、访谈、调研或其他可核对路径 |
| 状态 | `open`、`collecting`、`verified`、`deferred` 或 `accepted-residual` |
| 延期 / 复核 | 若为 `deferred`，记录延期理由、责任人、下一复核日期或触发条件 |
| 证据 / 结论 | 路径、决策号、审计号或明确的待确认说明 |

登记不是“已经知道一切”的声明。新发现的信息可追加；已被推翻的假设应保留其历史并记录后续决定。`required` 与 `non-blocking` 必须由目标的风险、不可逆性、验收、资源、安全、合规和依赖影响判断，不能只凭是否“暂时方便”决定。

### 设立与阶段门禁

1. **设立许可**：只要目标意图、初始边界、父级与最小可验证方向足以陈述，目标可以创建为 `draft` 或 `active`，即使信息表仍有开放项。若连这些最小信息都不能陈述，应先建立以澄清为交付物的探索目标或保持草稿，不得伪造完整方案。
2. **规划门禁**：影响范围、验收、资源、安全、合规或关键依赖的 required 信息，在方案冻结或承诺不可逆实施前必须验证。尚未验证时，允许规划“如何收集信息”，不允许把未经验证的假设当成实施承诺。
3. **实施门禁**：实施受某一 required 信息项影响的范围前，该项必须有证据。经用户裁决并记录的有界实验只能进入其明确的**信息收集范围**，该信息项保持 `collecting`，不等同于验证通过，也不得放行实验范围之外的实施。实施中发现关键未知时，暂停受影响范围、记录事实，并回流到信息表、决策或路线图。
4. **关门门禁**：任何影响成功标准、验收或风险主张的 required 信息项未闭环时，不得将目标标为 `done`；审计必须对照信息表核对状态与证据。

### 残余风险与用户裁决

用户可以接受残余风险，但编排器不得静默推断。接受记录必须写入 `01-decision.md` 或 `03-audit.md` 响应节，至少说明：未知是什么、影响范围、为何可接受、适用期限、缓解/监控方式、责任人或复审触发条件。`accepted-residual` 仅解除所写明范围内的门禁，不等同于 `verified`，也不改写未知本身的事实状态。信息冲突、是否以实验替代完整验证或是否接受残余风险，按 P-004 询问用户并留痕。

`deferred` 不得成为“以后再说”的同义词：必须保留原级别并记录延期理由、责任人和下一复核日期或触发条件。到达最晚需要阶段时，未获用户 residual 接受的 `deferred required` 自动按开放 required 处理并阻断受影响门禁；`deferred non-blocking` 仍须在下一个复核点重新评估。

### 子目标拆分

“先识别需要什么信息”与“收集信息”是两类有效工作，但不是每个目标的固定两个子目标。先维护信息表与阶段门禁；当澄清或收集工作有独立范围、依赖、产物证据、持续时间或并行价值时，按 P-001 创建“信息澄清/验证”或“信息收集”子目标。低风险、可逆且能在同一小目标内闭环的项，保留在原目标的信息表和检查点中。

**价值**：把未知本身变成可治理对象，既避免过早承诺，也避免用“以后再说”绕过规划、实施或关门的证据要求。

## 补充协议：工作区与共享资料引用

- 工作区是 Root Goal 与 canonical 范围的上下文，不是新的目标层级；同一项目的串行阶段继续由路线图和子目标承接。
- 共享资料固定引用只提供可追溯来源，不能替代用户确认、审计证据、P-004 裁决或 P-005 门禁。
- 工作区绑定或资料引用不完整时，受影响的读取、推理、写入和放行应 fail closed；物理资料存储、用户 CRUD、AI 访问执行和 Web 安全模型留给相应消费适配器的验证门禁。
